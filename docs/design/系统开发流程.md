# PSA 开发流程

> 适用团队：5 人（业务 Admin：都业梁；IT Admin：江勇超；全栈开发：刘浩之、刘富之、吴云杰）
> 开发节奏：日更或一日多更，小步快跑
> 仓库结构：monorepo（psa-backend / psa-front-desk-pc / psa-mini / psa-mini-program-new）

---

## 一、角色与职责

### 业务 Admin（都业梁）

负责需求的定义和验收。日常工作包括：

- 撰写需求 Issue（业务场景 + **验收标准** + **边界场景清单**）
- 在 dev 环境对每个 PR 完成验收，通过后开放生产部署
- 判断需求的优先级和拆分粒度，每个 Issue 应该是可独立交付、可独立验收的最小业务单元（**单 Issue 预估 ≤ 1 天，超出必拆**）

**关于 AI 工具的使用边界**：

都业梁作为团队中 AI 资源最充足的角色（Codex + Claude），将 AI 用在它最擅长的事上——**穷举边界场景和反例数据**，而不是猜代码位置。

1. **不写技术方案初稿**：不贴代码片段、不指定要改的文件和方法。技术方案由 IT Admin / 开发负责，避免 AI 猜路径、猜方法名带来的双花成本和责任边界模糊
2. **专注"业务侧反例"**：用 AI 推演"这个需求可能漏掉的 10 个边界场景"，把这些写进 Issue 的验收标准里。例如：跨月提交、跨年提交、单日 0 工时行、驳回后 BPM 状态未同步等
3. **准备测试数据**：把上述边界整理成可复现的反例数据，验收时直接用

### IT Admin（江勇超）

兼任 Tech Lead 和高级开发。日常工作包括：评审 Issue 的技术可行性并出技术方案、Review **复杂或破坏性 PR**、维护 CI/CD 流水线和 dev 环境、判断 PR 是否需要打 `breaking` 标签（用于客户提前知会）、执行生产部署。IT Admin 也参与编码，尤其是架构层面和跨模块的改动。

为避免单点瓶颈，**简单 PR（< 100 行 且 不涉及 migration / 跨模块）由两位开发互相 Review 即可合并**，IT Admin 只兜底复杂改动。

### 全栈开发（刘浩之、刘富之、吴云杰）

负责具体的功能开发和测试编写。日常工作包括：根据 Issue 创建 feature 分支并编码、编写对应的单元测试、本地自测通过后提 PR、根据 Review 意见修改代码。**两位开发互相 Review 简单 PR 并合并**，复杂改动交由 IT Admin Review。

## 二、版本定义

采用语义化版本（SemVer）：`主版本.次版本.补丁号`，例如 `1.3.2`。

**核心原则**：采用轻量混合分支模型，`main` 对应 prod 环境，`develop` 对应阿里云 dev 集成环境。日常小版本 `patch` 直接 PR 到 `main`；用户可感知的新功能、流程变化或破坏性改动（`feature` / `breaking`）先 PR 到 `develop` 做 dev 集成验收，验收通过后再通过 `develop -> main` release PR 发布生产。

### 自动版号规则

合并到 main 时根据改动类型自动打 git tag（CD 待配置）：

- 日常 `patch`（修复 / 文档 / 配置类）：补丁号 +1（如 v1.3.2 → v1.3.3）
- `feature`（用户可感知新功能）：次版本号 +1，补丁号归零（如 v1.3.2 → v1.4.0）
- `breaking`（破坏性变更）：主版本号 +1（架构大改或不兼容变更，当前阶段罕用）

> 改动类型由 PR 标题前缀（`fix(...)` / `feature(...)` / `chore(...)` 等）或分支名前缀（`fix/` / `feature/`）决定，**不是单独的 GitHub Label**。

### `feature` / `breaking` 改动类型的用途

这两类改动会决定是否需要进入 dev 集成环境：

1. **发布路径**：`patch` / `fix` / `chore` 类直接 PR 到 `main`；`feature` / `breaking` 类先 PR 到 `develop`，dev 验收通过后再 `develop -> main`
2. **客户知会**：`feature` / `breaking` 表示用户能感知的改动（新功能、流程变更），需在企业微信群提前告知或附发布说明
3. **版本号语义清晰**：让客户和团队从版本号就能看出这是日常修复还是新功能

### 谁决定改动类型

PR 提交者根据 Issue 类型在标题前缀中体现（`feature(scope):` / `fix(scope):` / `chore(scope):` 等），IT Admin 在 Review 时校正。**migration 涉及业务表结构变更默认按 `feature` 类处理**（强制走完整观察期）。

## 三、完整开发流程

### 第一步：业务 Admin 提 Issue

所有 GitHub 操作（Issue、PR、评论、标签、关闭等）必须使用 `gh` CLI，不使用网页手动操作。

**Issue 标题格式**：`{type}({scope}): 简短中文描述`

- `type` 取 `fix` / `feature` / `chore` / `docs` / `refactor` / `discuss` 之一
- `scope` 只允许小写字母 / 数字 / 连字符，例如：`feature(timesheet): 工时提交增加跨项目校验`

**Issue 正文必须包含**：`## 问题点`、`## 业务背景`、`## 目标`、`## 验收标准`、`## 边界场景清单（AI 辅助）`、`## 优先级建议`

**Issue 创建三要素**（缺一不可，详见 [CLAUDE.md §7](../../CLAUDE.md)）：

| 要素 | 值域 | 说明 |
|---|---|---|
| **优先级标签** | `P0` / `P1` / `P2` / `P3` | `gh issue create --label P2 ...` |
| **项目归属** | `PSA` 看板 | `gh issue create --project "PSA" ...` |
| **Issue Type** | `Bug` / `Feature` / `Task` | 创建后 GraphQL API 补设 |

边界场景清单建议借助 Codex / Claude 推演——跨月/跨年、空数据、异常状态、权限边界、并发提交等，每条写成"输入 → 期望输出"的可验证形式。测试数据（可选）把高风险边界整理成可复现的反例数据，标记为"验收必跑"。

**业务 Admin 不写技术方案**——不指定文件路径、不贴代码片段。技术方案由 IT Admin / 开发负责。

如果一个需求预估超过 1 天，业务 Admin 应该拆成多个 Issue，每个可以独立交付和验收。

### 第二步：IT Admin 评审与分配

IT Admin 收到 Issue 后做三件事：

1. **技术可行性评估**：判断改动涉及的模块、复杂度、与现有架构的衔接（framework 封装、BPM 状态同步等），评估工作量。复杂需求在 Issue 评论中给出技术思路骨架
2. **确认改动类型**：默认 `patch`（修复/文档/配置），新功能或流程变更为 `feature`，破坏性变更为 `breaking`；migration 涉及业务表结构变更默认 `feature`。类型体现在 PR 标题前缀，不是单独 Label
3. **分配**：指定给某位开发，或自己做。简单 bug 修复不需要评审，直接分配即可

### 第三步：开发编码

开发人员按改动类型选择基线分支：

- `patch` / 日常小版本：从最新 `main` 创建 `fix/issue-编号-简述` 或 `feature/issue-编号-简述`，完成后 PR 到 `main`
- `feature` / `breaking`：从最新 `develop` 创建 `feature/issue-编号-简述`，完成后 PR 到 `develop`，dev 验收通过后再 `develop -> main`
- P0 hotfix：从 `main` 创建 `hotfix/issue-编号-简述`，修复后 PR 到 `main` 并同步回 `develop`

分支名属于协作可读性规范，CI 只做提醒，不作为合并硬门禁；真正影响可追溯性的 Issue 关联、label、CI、Review 作为硬门禁处理。

编码过程中遵循项目约定：后端 Controller 分 admin/app 两套、公共能力放 framework 或 api 模块、业务实现放 biz 模块、改了业务逻辑的 PR 必须带对应的单元测试。

### 第四步：开发自测（Stage 1）

代码写完后，推送之前在本地做三件事：

1. **编译检查**：后端 `mvn clean compile -DskipTests`、前端 `npm run build`，确认能编译通过
2. **跑相关测试**：只跑自己改动涉及的单元测试，例如 `mvn test -pl yudao-module-fst/yudao-module-fst-biz -Dtest=TimeSheet75HourRuleTest`，不需要跑全量
3. **手动烟雾测试**：在本地环境手动走一遍主路径，改了驳回逻辑就操作一次驳回重提，看结果是否符合预期

三步完成后推送到远程分支，在 GitHub 上创建 Pull Request，关联对应的 Issue。PR 描述按关闭方式二选一：内部技术类、文档类、CI 类且“合并即完成”的 Issue 写 `closes #42`；需要发起人或业务 Admin 验收的 Issue 写 `refs #42`，验收通过后再由发起人手动关闭。目标分支按 label 决定：`patch` 默认 PR 到 `main`；`feature` / `breaking` 默认 PR 到 `develop`，dev 验收通过后再创建 `develop -> main` 的 release PR。

### 第五步：CI 自动检查（Stage 2）

PR 创建后 GitHub Actions 自动触发 `pr-gate.yml`，全程无需人工干预。CI 包含以下检查：

- **路径过滤**：自动检测本次 PR 改了哪些目录，只跑相关的检查。只改了前端就不跑后端编译，反之亦然
- **后端编译**：`mvn clean compile -DskipTests`
- **后端 P0 单元测试**：跑 4 个核心业务规则测试（7.5h 校验、重复提交拦截、驳回重提、取消后可编辑），加上 PR 中新增的测试
- **Flyway 迁移验证**：如果改了 migration 脚本，在 Testcontainers 空库上验证能从头跑通
- **前端构建 + Lint**：`npm run build` + `npm run lint`
- **小程序构建**：uni-app 编译检查（**初期标记 `continue-on-error`，工具链稳定后再卡门禁，避免 Day 1 阻塞**）
- **前后端状态契约校验**：检查后端枚举和前端常量是否同步

所有检查通过显示绿色对勾，任一失败显示红色叉号。失败时开发自行修复并重新推送，CI 自动重跑。同一个 PR 连续推送会自动取消上一次还在跑的检查。

目标：全部检查在 10 分钟以内完成。

### 第六步：PR Review（Stage 3）

CI 全绿是合并的必要条件，但不是充分条件。**Review 分两路**：

- **简单 PR**（< 100 行 且 不涉及 migration / 跨模块 / 核心业务规则）：两位开发互相 Review 即可合并，IT Admin 不卡位
- **复杂 PR**（migration、跨模块、核心业务规则、`feature` / `breaking` 标签）：IT Admin Review

人工审查关注要点：

- **代码质量**：有没有明显的逻辑漏洞、性能隐患、安全风险（SQL 拼接、硬编码、空指针）
- **数据库脚本**：migration 脚本是否安全。**DDL 变更必须可逆**——`add` 与 `drop` 绝不在同一 PR 里完成（先一个版本 add，下一个版本再 drop），保证可灰度回滚
- **测试覆盖**：改了业务逻辑的 PR 是否带了对应的单元测试（单向棘轮原则：只加不减）
- **跨模块一致性**：改了后端 API 的 PR，前端是否同步更新了
- **改动类型**：确认 PR 标题前缀（`fix` / `feature` / `chore` 等）与实际改动性质一致，决定目标分支是 `main` 还是 `develop`

### 第七步：按版本类型合并

**patch（日常小版本）**：

1. PR 直接合并到 `main`
2. IT Admin 部署生产（CD 待配置；配置完成前手动部署）
3. 如果 PR 使用 `closes #Issue`，GitHub 会在合并后自动关闭 Issue；如果 PR 使用 `refs #Issue`，业务 Admin 抽查关键路径，确认无环境差异问题后手动关闭 Issue

**feature / breaking（功能或流程变化）**：

1. PR 先合并到 `develop`
2. 部署 `develop` 当前代码到阿里云 dev 环境（CD 待配置；配置完成前由 IT Admin 手动部署 dev）
3. 在企业微信群推送部署通知，包含 PR 链接、`develop` commit、`@都业梁` 提示验收

### 第八步：业务 Admin 在 dev 环境验收（Stage 4）

`feature` / `breaking` 必须在 dev 环境通过业务 Admin 验收后才能进入 `main`。`patch` 不强制走 dev，但仍要在生产部署后抽查关键路径。

- 业务 Admin 按 Issue 中的验收标准 + 边界场景清单在 dev 环境验证
- 通过：在 Issue 中标记 `acceptance: passed`，并 @江勇超 创建或合并 `develop -> main` release PR
- 不通过：在 Issue 中反馈具体问题，开发立即修复，走新一轮 PR（dev 上的旧版本被下一次 dev 部署自动覆盖）

**`feature` / `breaking` 标签的 PR**：业务 Admin 需完整走一遍跨模块主流程（如工时提交 → 审批 → WIP 计算 → 会计凭证），重点验证集成衔接，验收时间预留更长。

### 第九步：feature / breaking 发布生产

收到业务 Admin 的"验收通过"信号后，IT Admin 合并 `develop -> main` release PR，再部署生产（CD 待配置；配置完成前由 IT Admin 手动部署）。生产部署后业务 Admin 抽查关键路径，确认无环境差异问题；需要人工验收关闭的 Issue 由发起人或业务 Admin 手动关闭。

**`feature` / `breaking` 标签的 PR**：部署前在企业微信群发布发布说明，告知客户即将上线的功能或行为变化。

## 四、测试策略

### 总体原则

把检查的压力从人转移到机器。开发人员只做最基本的自检，核心业务规则的守护交给 CI 自动化。4 人团队不追求测试覆盖率，只守住核心业务规则不被破坏。

### 单元测试

使用纯 Mockito 单元测试（`@ExtendWith(MockitoExtension.class)`），不启动 Spring 容器、不连数据库，单个测试毫秒级完成。测试文件本地和 CI 共用，写一次两处跑。

**P0 测试场景（4 个，首批编写）**：

| 场景 | 测试类 | 守护的业务规则 |
|---|---|---|
| 7.5h 校验 | TimeSheet75HourRuleTest | 日工时总和不足 7.5h 不可提交 |
| 重复提交拦截 | TimeSheetDuplicateSubmitTest | 同人同日不可重复提交 |
| 驳回重提 | TimeSheetRejectResubmitTest | 驳回后可修改被驳回行并重新提交 |
| 取消后可编辑 | TimeSheetCancelEditableTest | 取消后工时单回到可编辑状态 |

**日常新增测试**：每次改了业务逻辑的 PR 必须带对应的测试（单向棘轮原则）。IT Admin 在 Stage 3 Review 时检查这一点。

### 集成测试

`feature` / `breaking` 合并到 `develop` 后部署到 dev，由业务 Admin 在 dev 环境手动验收（见"第八步"），并完整走一遍跨模块主流程（工时提交 → 审批流程 → WIP 计算 → 会计凭证）。日常 `patch` 不强制进入 dev，验收聚焦 PR 自身的验收标准 + 生产部署后的关键路径抽查。端到端验证通过 Playwright MCP 在开发阶段按需执行（Claude Code 会话内启动浏览器，零 spec 文件维护成本）；`@playwright/test` 持久化 CI spec 因 dev ECS 资源限制（2 核 4G）暂缓，不写入 CI 流水线。

dev 环境部署在阿里云，一台低配 ECS（2 核 4G），SQL Server 同实例不同 database，Nacos/Redis/RocketMQ 与生产共用实例但用不同 namespace 隔离。

### 每日巡检

GitHub Actions 定时任务（`daily-patrol.yml`），每天早晨 08:30 自动运行（早于 09:00 上班节点，避免和反馈、Review 时段抢人），结果推送企业微信群。检查项包括：

- 生产数据状态分布漂移检测
- 主表/明细表/BPM 扩展表状态一致性扫描
- 全量 ESLint + 后端 Checkstyle
- Schema drift 检测（生产库 vs migration 脚本）

巡检不阻断开发流程，发现问题由 IT Admin 判断优先级后转为 Issue。

## 五、CI/CD 配置

### PR 门禁（pr-gate.yml）

```
开发推代码到 feature 分支
      │
      ▼
  创建 / 更新 Pull Request
      │
      ▼
  GitHub Actions 自动触发
      │
      ▼
  路径过滤：检测改了哪些目录
      │
      ├── psa-backend/ 有变更 → 后端编译 + P0 测试 + Flyway 验证
      ├── psa-front-desk-pc/ 有变更 → 前端构建 + Lint
      ├── psa-mini-program-new/ 有变更 → 小程序构建
      └── 状态枚举有变更 → 契约校验
      │
      ▼
  gate 汇总 job
      │
      ├── 全部通过 ✅ → PR 可合并
      └── 任一失败 ❌ → PR 不可合并，开发修复后重推
```

分支保护规则：GitHub Settings → Rulesets / Branches，对 `develop` 和 `main` 都启用 PR 保护，要求 `PR Gate` 与 `PR Hygiene` 通过后才可合并。

### CD 自动版号（待配置）

合并到 main 时根据 PR 标题前缀（改动类型）自动打 git tag（CD 待配置）：

- `fix` / `chore` / `docs` / `refactor`（日常 patch 类）→ 补丁号 +1（如 v1.3.2 → v1.3.3）
- `feature` → 次版本号 +1，补丁号归零（如 v1.3.2 → v1.4.0）
- `breaking` 类（PR 正文标注）→ 主版本号 +1（如 v1.3.2 → v2.0.0）

### 部署流程（CD 待配置）

按 label 分两条路径：

```
patch PR
      │
      ▼
  PR 到 main
      │
      ▼
  PR Gate + PR Hygiene + Review
      │
      ▼
  合并 main → 部署 prod（CD 待配置；当前可手动）
      │
      ▼
  业务 Admin 抽查关键路径 → 关闭 Issue
```

```
feature / breaking PR
      │
      ▼
  PR 到 develop
      │
      ▼
  PR Gate + PR Hygiene + Review
      │
      ▼
  合并 develop → 部署 dev（CD 待配置；当前可手动）
      │
      ▼
  业务 Admin 在 dev 上完整验收
      │
      ├── 不通过 → 反馈问题 → 开发修复 → 新 PR 到 develop
      │
      ▼
  通过 → release PR: develop → main
      │
      ▼
  合并 main → 部署 prod（CD 待配置；当前可手动）
      │
      ▼
  业务 Admin 抽查关键路径 → 关闭 Issue
```

**`feature` / `breaking` 标签**：部署 prod 前在企业微信群发布发布说明告知客户。

## 六、分支策略

采用 `main` + `develop` + feature branches 模式。

- `main`：prod 环境对应分支，承接日常 `patch` PR，以及 dev 验收通过后的 release PR
- `develop`：阿里云 dev 集成环境对应分支，只承接 `feature` / `breaking` 这类需要集成验收的 PR
- `feature/issue-{编号}-{简述}` / `fix/issue-{编号}-{简述}`：推荐开发分支命名；`patch` 从 `main` 创建并 PR 回 `main`，`feature` / `breaking` 从 `develop` 创建并 PR 回 `develop`
- `hotfix/issue-{编号}-{简述}`：P0 hotfix 可从 `main` 创建，合并 `main` 后必须同步回 `develop`
- 不使用 release 分支——4 人团队没有并行发版的需求

PR 合并策略：Squash and merge，保持 `develop` 和 `main` 历史干净。每个 PR 对应一个 commit，commit message 包含 Issue 编号。

## 七、环境说明

| 环境 | 用途 | 部署方式 |
|---|---|---|
| local | 开发人员本地调试 | 各服务 `application-local.yaml`，profile=local |
| dev | **功能集成 + 业务 Admin 验收**（对应 `develop`；`feature` / `breaking` 合并后部署） | 阿里云独立 ECS，独立数据库，共用 Nacos/Redis/RocketMQ（不同 namespace） |
| prod | 生产环境（对应 `main`；`patch` 直接发布，`feature` / `breaking` 由 `develop -> main` 发布） | 阿里云 |

dev 是功能和流程变化的集成验收环境；日常 `patch` 不强制经过 dev，但必须通过 CI、Review 和生产关键路径抽查。客户不应成为第一个看到功能级或流程级 bug 的人。

## 八、Issue 与 PR 的关联规范

### Issue 规范

- Issue 标题格式：`{type}({scope}): 简短中文描述`；正文必须包含 `## 问题点`、`## 业务背景`、`## 目标`、`## 验收标准`、`## 边界场景清单（AI 辅助）`、`## 优先级建议`
- Issue 创建三要素（优先级标签 P0-P3、项目归属 PSA 看板、Issue Type）缺一不可，详见 [CLAUDE.md §7](../../CLAUDE.md)
- 如果一个 Issue 确实需要拆成多个 PR，在 Issue 中用 checklist 跟踪进度

### PR 规范

**PR 标题格式**：`{type}({scope}): 简短中文描述 (#IssueNumber)`

- `type` 必须是 `fix` / `feature` / `chore` / `docs` / `refactor` / `discuss` 之一（小写，pr-hygiene 正则硬卡）
- `scope` 只允许小写字母 / 数字 / 连字符，不能用中文 / 下划线 / 空格
- 示例：`fix(timesheet): 工时提交增加跨项目校验 (#42)`

**PR 正文必须包含**：`## 概要`、`## 改动范围`、`## 关联 Issue`（`closes #N` 或 `refs #N`）、`## Test Plan`（至少一条已勾选项）

**关联语义**：

- `closes / fixes / resolves #N`：合并即完成，无需业务验收（技术类、CI 类、文档类）
- `refs / relates to #N`：合并后仍需业务 Admin 在 dev / 生产验收后手动关闭

**判断口诀**：合并后是否还需要用真实业务数据 / 流程在 dev 或生产验证一次？需要 → `refs`，不需要 → `closes`。

### 自动化执行口径

| 类型 | 规则 | 执行方式 |
|---|---|---|
| 硬门禁 | PR 必须带一个**优先级标签** `P0` / `P1` / `P2` / `P3`（创建时显式传 `--label P{N}`） | `PR Hygiene` 失败则不可合并 |
| 硬门禁 | PR 标题格式 `{type}({scope}): 描述 (#N)`，type 限小写枚举（同时决定改动类型与目标分支） | `PR Hygiene` 失败则不可合并 |
| 硬门禁 | PR 描述包含 `## Test Plan`，且至少勾选一项已执行检查（`- [x]`） | `PR Hygiene` 失败则不可合并 |
| 硬门禁 | PR 描述包含 `closes #N` / `refs #N` 等 Issue 关联语句 | `PR Hygiene` 失败则不可合并 |
| 硬门禁 | `PR Gate` 编译、测试、构建、Lint 通过 | GitHub branch protection / ruleset 要求 |
| 硬门禁 | 复杂 PR 经过 IT Admin Review；所有 PR 通过至少一次 Review | GitHub branch protection / CODEOWNERS |
| 硬门禁 | 合并方式使用 Squash merge | GitHub repository settings |
| 推荐规范 | 分支名使用 `feature/issue-{N}-{简述}` / `fix/issue-{N}-{简述}` / `hotfix/issue-{N}-{简述}` | `PR Hygiene` warning，不阻断 |
| 推荐规范 | 本地自测记录完整、风险项说明清楚 | PR Template 引导，Review 时人工判断 |

## 九、客户反馈处理流程

### 反馈渠道

客户反馈统一通过**企业微信群**进入。不在群里的反馈（电话、口头、邮件）由接收人转录到群里，确保所有反馈都有记录可追溯。

### 分诊机制

都业梁和江勇超共同盯群，按问题类型分工接手：

| 问题类型 | 接手人 | 典型表现 |
|---|---|---|
| 业务问题 | 都业梁 | 流程怎么走、数据为什么不对、新功能咨询、配置问题 |
| 技术问题 | 江勇超 | 系统报错、登录异常、页面打不开、性能问题 |

接手原则：先到先接，5 分钟内响应"收到，正在看"。判断不清归属时由接手人继续跟进，不做二次转手。如果发现是另一类问题，在群里 @对方协助，但工单的跟进责任不转移。

### 处理路径

接手后按以下分支处理：

1. **使用问题/咨询**：直接在群里答复，10 分钟内闭环
2. **配置问题**：自己调整或指导客户调整，处理完后在群里确认
3. **疑似 Bug**：要求客户提供复现步骤、截图、时间点，在 GitHub 上创建 Issue（label: `bug`），按下面的优先级机制处理
4. **新需求**：在 GitHub 上创建 Issue（label: `enhancement`），由都业梁评估优先级后纳入开发计划

### Bug 优先级机制

所有生产 bug 按严重程度分为三级，开发响应节奏不同：

| 级别 | 定义 | 响应时间 | 操作 |
|---|---|---|---|
| **P0** | 阻断核心业务，多数用户受影响（无法登录、工时无法提交、审批流崩溃、数据错乱） | 立即 | 开发停下手头工作立即修，hotfix PR 合入 `main` 部署生产；如 `develop` 存在未发布功能，需 cherry-pick 或补 PR 同步，事后补流程 |
| **P1** | 影响部分功能或少数用户，有 workaround | 当天 | 开发完成手头 PR 后立即接手，当天发版 |
| **P2** | 体验问题、非核心功能、边缘场景 | 本周 | 纳入正常开发队列，按节奏处理 |

**P0 的特殊流程**：可以跳过 dev 观察期、跳过业务 Admin 验收先上线止血，但 **江勇超的 Code Review 不能跳**（哪怕只是 5 分钟扫一眼，挡掉低级错误），必须在 24 小时内补一个 follow-up Issue 复盘和加测试用例，避免再犯。江勇超负责把关 P0 hotfix 的代码质量。

### 客户反馈与 Issue 的关联

每条转成 Issue 的反馈，在 Issue 描述中保留：客户名/反馈时间/原始描述/复现步骤。客户反馈类 Issue 的 PR 使用 `refs #Issue`，避免代码合并后自动关闭；Issue 解决、部署生产并完成客户或业务侧确认后，在企业微信群回复客户结果，再手动关闭 Issue。这样客户反馈和代码改动有完整的可追溯链路。

## 十二、架构治理与平台规划

### 12.1 为什么需要单独的"架构治理"

baseline 文档（`docs/baseline/`）已经识别出一批**平台级议题**——数据骨架、状态机骨架、安全骨架、运维骨架。这些议题的特点是：

- **不来自客户 Issue**：客户报的是"工时取消按钮没生效"，但底下的根因可能是"全平台没有统一状态机"
- **横切多个域**：状态事件表、月结锁账、密钥治理这类议题影响所有业务模块
- **风险驱动而非需求驱动**：风险不出问题不会有人提，但一出问题代价极高（数据错乱、月结失败、密钥泄露）

如果只走"客户提需求 → 开发"的链路，这些议题永远排不上队。必须由 IT Admin **主动立项**并单独排期。

### 12.2 角色与职责（架构层）

| 角色 | 职责 |
|---|---|
| **江勇超（IT Admin / 架构规划者）** | **PSA 平台架构的唯一规划者**。负责从 baseline 提取议题、定优先级、出技术方案骨架（状态机定义、表结构、CI 规则等），主导密钥/基础设施类不可代理的治理 |
| **都业梁（业务 Admin）** | 在涉及**核算口径**的架构议题上参与评审：月结锁账规则、状态机的状态定义（特别是与《权责发生制经济事实核算指导文件》对齐的部分）。其他纯技术议题不参与 |
| **刘浩之、刘富之、吴云杰（全栈开发）** | 按域分工承接架构 Issue 的实施（见 12.5）。一人主导一个状态机域的 StateService 落地，江勇超把关方案与代码 |

### 12.3 架构 Issue 的立项与节奏

- **立项节奏**：每月初江勇超从 baseline 07 / 08 / 10 中挑 1-2 件平台级议题，作为**架构 Issue** 立项，label: `architecture`
- **产能分配**：业务 Issue 与架构 Issue 共享开发产能，建议比例 **70% 业务 / 30% 架构**。架构 Issue 不能被业务 Issue 长期挤压（连续 2 个月没排上必复盘）
- **`architecture` 与版本标签共存**：架构 Issue 同时打 `architecture` + 版本标签（`patch` / `feature` / `breaking`）。一个补 PK 的 Issue 是 `architecture` + `patch`；一个引入状态事件表是 `architecture` + `feature`
- **验收方式**：架构 Issue **没有客户验收**，**由江勇超在 dev 环境验收**（不需要都业梁验收，除非涉及核算口径）。验收标准在 Issue 里由江勇超自己写明
- **Issue 来源**：由江勇超主动创建。也允许开发在编码中发现根因性架构问题时建议立项，但是否立项由江勇超拍板

### 12.5 全栈开发按域分工

为了让两位开发的工作不再"看 Issue 派发"，而是有**自己长期负责的域**，按以下原则分工：

| 开发 | 主导域（业务侧重） | 兼修域 |
|---|---|---|
| **刘浩之** | **财务核算侧**：CarryForward / Project / CuValue | WIP 二次结转、合同条款金额计算 |
| **刘富之** | **业务流程侧**：Timesheet / Opportunity / Leads | 任务/资源确认、周会/项目过程控制 |

**分工含义**：

- 主导域内的架构 Issue（StateService 落地、数据骨架修复）默认分配给主导人
- 主导域内的业务 Issue 也优先分配给主导人，便于积累领域知识
- 跨域的复杂 Issue 由江勇超分配；P0/P1 bug 不论域归属，先到先接
- 域分工**不等于代码所有权**：两人都可改任意代码，但主导人在该域有"了解全貌"的责任，Review 时优先 @ 主导人

**不可代理的部分**：

- 数据骨架修复（PK / UK / FK / Flyway 落地）：可以交开发实施，但**生产 DDL 执行只能江勇超**
- 安全与密钥骨架：**江勇超独占**，不分配给开发
- 运维骨架（监控、备份、BPM 流程模型部署）：**江勇超独占**
