# 数据表分类标准设计

> 目的:为 FORCITIS 系统的业务数据表、分析数据表、治理数据表和审计日志表建立**统一的分类法 + 命名前缀**,让任何人看到表名就知道它属于哪一类、谁负责、能不能改、有没有历史。
> 状态:草案,待团队评审

---

## 一、为什么需要这份标准

### 1.1 当前问题

- 表名前缀混乱:`psa_` / `system_` / `infra_` 等混用,业务语义不统一
- 同一类数据散落多处:员工信息在 `system_users`、部门在 `system_dept`、岗位散落别处
- 主数据、配置、字典、业务表混在同一命名空间下,无法快速识别
- 新人入职无法快速理解数据架构
- 代码评审、外部审计、技术评审缺乏对话基础

### 1.2 这份标准解决什么

- 任何人看到纳入本标准的表名,3 秒识别出归属类别
- 创建纳入本标准的数据表时,有明确的归类依据
- 代码评审有检查清单
- 数据治理、审计、合规有共同语言

## 二、8 大数据表类别

### 2.1 完整分类表

| # | 类别 | 前缀 | 性质 | 是否可改 | 是否需要历史 |
|---:|---|---|---|---|---|
| 1 | OLTP 业务表 | `psa_` | 业务事件 | ✅ 频繁改 | 部分需要 |
| 2 | 主数据 | `mdm_` | 共享实体/标准 | 🔶 低频改 | ✅ 必须 |
| 3 | OLAP 事实 | `fact_` | 分析事实 | ❌ INSERT-only | ✅ 完整 |
| 4 | OLAP 维度 | `dim_` | 分析维度 | 🔶 新增版本 | ✅ 完整 |
| 5 | 参考数据/字典 | `ref_` | 枚举常量 | 🔶 极少改 | ❌ 不需要 |
| 6 | 规则 / 控制 | `rule_` | 行为规则参数、控制开关 | 🔶 业务调整时 | 通常需要 |
| 7 | 元数据 | `meta_` | "数据的数据" | 🔶 治理时 | 需要 |
| 8 | 审计日志 | `audit_` / `log_` | 操作痕迹 | ❌ INSERT-only | ✅ 永久 |

注:本标准只覆盖业务、分析、治理和审计相关的数据表。数据库引擎表、中间件表、框架自带表不进入 8 大类；这类表保留框架原名,在部署清单中说明来源和用途。

### 2.2 各类详细规范

---

#### 类别 1:OLTP 业务表 — 前缀 `psa_`

**定义**:业务过程在"发生瞬间"产生的、需要被业务流程处理的数据。

**判定标准**(满足任意一条即归入此类):
- 有明确的"创建时刻"和"提交时刻"
- 有状态机(草稿→提交→审批→完成)
- 业务用户直接通过页面录入
- 高频写入

**设计原则**:
- 通常按 3NF 范式
- 必须有 `created_time` / `updated_time` / `created_by` / `updated_by`
- 软删除用 `deleted` 字段
- 状态用枚举,值定义在 `ref_xxx_status` 表

**典型例子**:

| 当前表名 | 标准化后 |
|---|---|
| `psa_time_sheet` | `psa_time_sheet` |
| `psa_time_sheet_works` | `psa_time_sheet_works` |
| `psa_posting_batch` | `psa_posting_batch` |
| (未来) | `psa_expense` |
| (未来) | `psa_invoice` |
| (未来) | `psa_payment` |

**反模式**:
- ❌ 把员工档案、客户档案放在 `psa_` 前缀下 → 应该是 `mdm_`
- ❌ 把审批历史放在 `psa_xxx_history` → 应该是 `audit_`

---

#### 类别 2:主数据 — 前缀 `mdm_`

**定义**:被多业务过程共享、相对稳定、是"标准/实体"而不是"事件"的数据。

**判定标准**(满足任意一条即归入此类):
- 修改频率低(通常按月、季度或年度调整)
- 被多个业务过程引用
- 需要权威维护(指定 owner)
- 必须保留历史(变更后历史核算要能回溯)

**设计原则**:
- 必须有版本管理字段:`effective_from` / `effective_to` / `is_current`
- 必须有 `version_no`(版本号)
- 修改通过新增版本,不覆盖
- OLTP 和 OLAP 都通过业务键引用,不复制属性
- 必须有 owner 字段(谁负责维护)

**典型例子**:

| 主数据 | 表名 | Owner |
|---|---|---|
| 员工 | `mdm_employee` | HR |
| 项目 | `mdm_project` | 项目管理办公室 |
| 客户 | `mdm_customer` | 销售 |
| 合同 | `mdm_contract` | 法务/销售 |
| 合同条款 | `mdm_contract_term` | 法务/销售 |
| 费率卡 | `mdm_rate_card` + `mdm_rate_card_detail` | HR / 财务 |
| 会计科目 | `mdm_account` | 财务 |
| 组织架构 | `mdm_org` | HR / 行政 |
| 工种/角色 | `mdm_labor_role` | HR |
| 期间定义 | `mdm_accounting_period` | 财务 |
| 工作日历 | `mdm_workday_calendar` | 财务 / HR |

**当前需要升级的表**:

| 当前 | 升级为 | 升级动作 |
|---|---|---|
| `system_users` | `mdm_employee` | 增加 Type 2 SCD 字段,部门/岗位历史保留 |
| `system_dept` | `mdm_org` | 增加层级、生效期 |
| `system_post` | `mdm_labor_role` | 增加历史版本 |

**反模式**:
- ❌ 主数据没有历史保留 → 员工调岗后,旧工时核算用了新部门
- ❌ 主数据散落在业务表的快照字段里 → 改一次几十处不一致

---

#### 类别 3:OLAP 事实表 — 前缀 `fact_`

**定义**:为分析、报表、决策服务的"业务过程已发生事件"的数据,经过 posting/确认后固化。

**判定标准**:
- 一次性 INSERT,永不 UPDATE
- 度量字段必须声明聚合口径(Additive / Semi-additive / Non-additive)
- 维度通过 `_sk` 代理键引用,不冗余属性
- 单一 Business Process,单一 Grain

**设计原则**:
- 表名 = `fact_` + BP 名称
- 必须有 grain 陈述句(写在表注释里)
- 所有外键 NOT NULL(用语义化空值行处理业务空)
- 必须声明表类型(Transaction / Periodic Snapshot / Accumulating Snapshot)

**典型例子**:

| 表名 | 表类型 | BP |
|---|---|---|
| `fact_time_sheet` | Transaction | 工时 posting |
| `fact_time_sheet_adjustment` | Transaction | 工时调整 |
| `fact_time_sheet_writeoff` | Transaction | 工时核销 |
| `fact_expense` | Transaction | 费用 posting |
| `fact_revenue` | Transaction | 收入确认 |
| `fact_invoice` | Transaction | 开票 |
| `fact_payment` | Transaction | 回款 |
| `fact_voucher` | Transaction | 凭证生成 |
| `fact_allocation` | Transaction | OH 分摊 |
| `fact_wip_snapshot` | Periodic Snapshot | WIP 月末快照 |
| `fact_balance_sheet_snapshot` | Periodic Snapshot | 资产负债月末快照 |

**反模式**:
- ❌ fact 表里有 `status` / `last_xxx_time` → 违反 Transaction Fact 不可 UPDATE 原则
- ❌ fact 表里有 `employee_code` / `project_name` 等维度属性 → 应该在 dim 表
- ❌ fact 表里有未声明聚合口径的 `cost_rate`(费率) → 费率通常应放在 dim 或作为快照证据并明确不可直接求和

---

#### 类别 4:OLAP 维度表 — 前缀 `dim_`

**定义**:为 fact 表提供"上下文"的描述性数据,通过代理键被 fact 引用。

**判定标准**:
- 主键是无业务含义的代理键(`_sk`,通常 bigint 自增)
- 同时持有业务键(原 OLTP 主键)用于回溯
- Type 2 SCD 维度:含 `effective_from` / `effective_to` / `is_current`

**设计原则**:
- 表名 = `dim_` + 维度名
- 主键命名 = `{维度名}_sk`(如 `employee_sk`)
- 业务键命名 = 原系统主键名(如 `person_id`)
- 反范式化:把多层级的属性 denormalize 到一张表
- Junk Dimension:多个低基数标志位合并成一张 dim

**典型例子**:

| 表名 | SCD 类型 | 来源 |
|---|---|---|
| `dim_date` | Type 1 | 预生成日历 |
| `dim_accounting_period` | Type 1 | mdm_accounting_period |
| `dim_employee` | Type 2 | mdm_employee |
| `dim_org` | Type 2 | mdm_org |
| `dim_bu` | Type 2 | mdm_org 派生 |
| `dim_project` | Type 2 | mdm_project |
| `dim_customer` | Type 2 | mdm_customer |
| `dim_contract_term` | Type 2 | mdm_contract_term |
| `dim_account` | Type 1 | mdm_account |
| `dim_rate_card` | Type 2 | mdm_rate_card |
| `dim_charge_type` | Type 1 | ref_charge_type |
| `dim_labor_role` | Type 2 | mdm_labor_role |

**反模式**:
- ❌ dim 表用业务主键当主键 → 主数据变更时引用断裂
- ❌ Type 2 dim 没有 `effective_from/to` → 历史快照无法实现
- ❌ 一个维度拆成多张 snowflake 表 → 查询性能下降

---

#### 类别 5:参考数据 / 字典 — 前缀 `ref_`

**定义**:枚举、字典、固定常量,通常永不变或几乎不变,数据量小。

**判定标准**:
- 几乎不变(通常多年稳定)
- 不需要历史(改了直接覆盖)
- 数据量小(几行~几百行)
- 全公司、全行业通用

**与主数据的区别**:

| 维度 | 主数据 `mdm_` | 参考数据 `ref_` |
|---|---|---|
| 变化频率 | 按月、季度或年度调整 | 多年稳定 |
| 历史保留 | 必须 | 不需要 |
| 数据量 | 中等 | 小 |
| 维护方 | 业务部门 | 系统管理员 |
| 例子 | 员工、客户 | 币种、国家 |

**设计原则**:
- 表名 = `ref_` + 概念名(单数)
- 必须有 `code`(业务编码,如 'CNY') + `name`(显示名)
- 必须有 `sort_order`(显示顺序)
- 不需要软删除,改用 `is_active`

**典型例子**:

| 表名 | 内容 |
|---|---|
| `ref_currency` | 币种代码(CNY/USD/EUR) |
| `ref_country` | 国家代码 |
| `ref_charge_type` | 工时类型(DELIVERY/PRESALE/OH) |
| `ref_billable_type` | 可计费类型 |
| `ref_data_origin` | 数据来源枚举 |
| `ref_approval_status` | 审批状态枚举 |
| `ref_posting_type` | posting 类型(TIMESHEET/EXPENSE/REVENUE等) |

**反模式**:
- ❌ 把枚举值硬编码进 Java 代码 → 加一种类型改 10 处
- ❌ 把所有 ref 塞进一张 `sys_dict` 表 → 查询复杂、性能差
- ❌ ref 表没有 `code` 字段,只有 `id` → 跨环境同步混乱

---

#### 类别 6:规则 / 控制 — 前缀 `rule_`

**定义**:系统行为参数、业务规则参数、业务控制开关。

**判定标准**:
- 改动会影响系统行为,不只是显示
- 通常配套管理后台界面
- 改动需要审批
- 不是"数据",是"规则"

**与参考数据的区别**:

| 维度 | 参考数据 `ref_` | 规则 / 控制 `rule_` |
|---|---|---|
| 性质 | 枚举常量 | 行为规则 |
| 影响 | 显示文本 | 系统行为 |
| 例子 | 币种代码 | "迟到入账阈值 = 5 天" |

**设计原则**:
- 表名 = `rule_` + 规则或控制主题
- 必须有 `effective_from` / `effective_to`(配置生效期)
- 必须有 `last_modified_by` / `last_modified_time`
- 关键配置必须配套审批流程
- 改动写入 `audit_config_change`

**典型例子**:

| 表名 | 用途 |
|---|---|
| `rule_posting_rule` | posting 引擎业务规则 |
| `rule_late_arrival_threshold` | 迟到入账阈值 |
| `rule_oh_allocation_method` | OH 分摊方法选择 |
| `rule_wip_aging_threshold` | WIP 账龄预警阈值 |
| `rule_feature_flag` | 特性开关 |
| `rule_time_sheet_period_control` | 工时期间采集、补齐、posting 控制 |

**反模式**:
- ❌ 把这些规则硬编码进代码 → 改一次发一次版
- ❌ 配置改动没有审计 → 业务出问题无法追溯

---

#### 类别 7:元数据 — 前缀 `meta_`

**定义**:"数据的数据"——描述其他表/字段/指标的信息。

**判定标准**:
- 不是业务数据
- 不是业务事件、主数据、规则、日志或框架基础设施数据
- 服务于数据治理、审计、血缘分析

**设计原则**:
- 表名 = `meta_` + 描述对象
- 前期可用 Excel/Confluence 管理,后期落库
- 必须有 owner

**典型例子**:

| 表名 | 用途 |
|---|---|
| `meta_table_catalog` | 纳入本标准的数据表目录:谁负责、什么含义、属于哪类 |
| `meta_column_dictionary` | 字段含义字典(业务术语 ↔ 物理字段) |
| `meta_data_lineage` | 数据血缘(fact_time_sheet 来自哪些上游) |
| `meta_data_quality_rule` | 数据质量规则定义 |
| `meta_business_metric` | **业务指标定义**(BU 毛利的标准公式)— 即"语义层" |

**对 FORCITIS 当前阶段的建议**:
- 不必专门做元数据表
- 用 Excel / Confluence / Notion 管理即可
- 团队人数超过 20 人 + fact 表超过 10 张时,再考虑落库

---

#### 类别 8:审计日志 — 前缀 `audit_` 和 `log_`

**定义**:记录"谁在何时做了什么"的痕迹数据。

**两个前缀的区别**:

| 前缀 | 用途 | 例子 |
|---|---|---|
| `audit_` | **业务审计**——业务实体的变更 | audit_data_change, audit_approval_history |
| `log_` | **技术日志**——系统运行的痕迹 | log_user_login, log_api_call |

**设计原则**:
- 一次性 INSERT,永不 UPDATE
- 必须有 `operator` / `operation_time` / `operation_type`
- 大表必须分区(按月或按日)
- 定期归档到冷存储

**典型例子**:

| 表名 | 内容 |
|---|---|
| `audit_data_change` | 数据变更审计(谁改了 mdm_employee 哪个字段) |
| `audit_posting_event` | posting 事件审计 |
| `audit_approval_history` | 审批历史 |
| `audit_config_change` | 配置变更审计 |
| `audit_late_arrival_fix` | 迟到数据 _sk 回填审计 |
| `log_user_login` | 登录日志 |
| `log_api_call` | API 调用日志 |
| `log_etl_run` | ETL 运行日志 |

**不纳入 8 大类的边界**:
- 聚合结果不单独设一类。若是可复算的消费口径,优先做视图或语义层；若需要物化且承载分析事实,按 `fact_` 快照/周期事实表评审。
- 基础设施/框架表不单独设一类。框架自带表(如 XXL-JOB、Flowable、消息队列、锁表、会话表)保留框架原名,不纳入业务数据表分类标准。

## 三、命名规范汇总

### 3.1 前缀对照

```
psa_      OLTP 业务表
mdm_      主数据
fact_     OLAP 事实表
dim_      OLAP 维度表
ref_      参考数据 / 字典
rule_     规则 / 控制
meta_     元数据
audit_    业务审计
log_      技术日志
```

### 3.2 字段命名约定

| 字段类型 | 命名规则 | 例子 |
|---|---|---|
| 主键(代理键) | `{表名}_sk` 或 `id` | `employee_sk`, `id` |
| 业务键 | 沿用业务命名 | `person_id`, `project_code` |
| 外键 | 引用表的主键名 | `employee_sk`, `project_sk` |
| 创建时间 | `created_time` | - |
| 创建人 | `created_by` | - |
| 更新时间 | `updated_time` | - |
| 更新人 | `updated_by` | - |
| 软删除 | `deleted`(bit) | - |
| Type 2 生效期 | `effective_from` / `effective_to` | - |
| Type 2 当前标记 | `is_current` | - |
| 启用标记 | `is_active` | - |
| 排序 | `sort_order` | - |

### 3.3 命名反模式

| 不要这样 | 改成 |
|---|---|
| `tbl_employee` | `mdm_employee`(不要 tbl 前缀,前缀必须表达类别) |
| `t_emp` | `mdm_employee`(不要缩写) |
| `employee_info` | `mdm_employee`(不要 _info / _detail / _data 后缀) |
| `sys_user_table` | `mdm_employee`(不要 _table 后缀) |
| `users` | `mdm_employee`(不要用复数) |
| `emp01` | `mdm_employee`(不要数字编号) |

## 五、落地路径

### 5.1 不必一次性改名

**改名涉及全栈代码改动,风险高**。建议:

| 阶段 | 动作 |
|---|---|
| 阶段 1(立刻) | **新建且纳入本标准的数据表必须遵循新前缀** |
| 阶段 2(随版本迭代) | 已有业务表可保留原名,但在内部文档中标注它们的归类 |
| 阶段 3(大版本) | 借助 v2.0 / v3.0 数据架构升级,逐步改名 |
| 阶段 4(长期) | 纳入本标准的老表逐步完成改名 |

### 5.2 文档要求

每张纳入本标准的数据表必须有以下元信息(放在 `meta_table_catalog` 或 Confluence):

| 字段 | 含义 |
|---|---|
| 表名 | 物理表名 |
| 类别 | 8 类之一 |
| 中文名 | 业务称呼 |
| 业务描述 | 1-2 句话说明 |
| Owner | 负责的人/部门 |
| 创建时间 | - |
| 最近变更时间 | - |
| 上游来源 | 数据从哪来 |
| 下游消费 | 数据被谁用 |

### 5.3 代码评审检查清单

新建且纳入本标准的数据表 PR 必须通过以下检查:

- [ ] 表名前缀是否符合 8 类规范?
- [ ] 是否有 owner?
- [ ] 是否填入 `meta_table_catalog`?
- [ ] 如果是 `mdm_`,是否有 Type 2 SCD 字段?
- [ ] 如果是 `fact_`,是否声明了表类型(Transaction/Snapshot)?
- [ ] 如果是 `rule_`,是否配套了变更审计?
- [ ] 字段命名是否符合 3.2 节约定?

## 备注

- 本标准 v1.0 是草案,需要团队评审后正式发布
- 建议先在新模块试运行 1-2 个月,再全面推广
- 标准本身随业务演进可能调整,但 8 类基本框架不应轻易动
