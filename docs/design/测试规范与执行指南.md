## 1. 总览与测试哲学

### 1.1 测试金字塔

```
                             ┌──────────────────────────────────┐
                             │  业务 Admin 手工验收（dev 环境）   │
                             │  合并到 develop 后，业务侧验收     │
                             └──────────────────────────────────┘
                        ┌──────────────────────────────────────────┐
                        │  开发手工测试（worktree dev server）       │
                        │  push 前，在 worktree 里跑冒烟             │
                        └──────────────────────────────────────────┘
                   ┌──────────────────────────────────────────────────┐
                   │  CI 自动测试（每次 push 到 PR 分支自动触发）        │
                   │  pr-gate.yml 全部 pass 才能合并                   │
                   └──────────────────────────────────────────────────┘
              ┌──────────────────────────────────────────────────────────┐
              │  本地单元测试（在 worktree 内，commit 前运行）              │
              │  纯 Mockito，毫秒级，改业务逻辑必须附带，单向棘轮           │
              └──────────────────────────────────────────────────────────┘
```

### 1.2 各层与 Worktree 工作流的对应关系

| 层级 | 在哪里跑 | 触发时机 | 是否阻断合并 |
|---|---|---|---|
| 单元测试 | worktree 内 + CI | commit 前手跑 + 每次 push | 是（CI 红不能合并） |
| 构建 / Lint | worktree 内 + CI | push 前手跑 + 每次 push | 是 |
| 开发手工测试 | worktree dev server | push 前 | 是（开发自我要求） |
| 多 issue 合测 | integration worktree | PR 互相依赖时 | 否（临时验证） |
| dev 环境验收 | develop 分支部署后 | PR 合并后 | 由业务 Admin 关 Issue |

### 1.3 核心原则：改了业务逻辑必须附带单元测试（单向棘轮）

**单向棘轮**：项目测试覆盖只增不减。提交业务逻辑变更的 PR 必须在同一 PR 内附带对应单元测试，否则 PR Review 阶段应退回。

判断"是否属于业务逻辑"的口诀：

- 有 if/switch 分支判断业务状态 → 必测
- 有数字计算（金额、工时、比率）→ 必测
- 有权限/状态守卫（谁能做什么、什么状态下才允许）→ 必测
- 纯 CRUD（`save`/`getById`/`listAll` 无额外逻辑）→ 不必测

> **⚠️ "纯 CRUD 不必测"的边界**：指**业务逻辑**层面无分支判断。
> - **Mapper SQL** 即使是简单查询，只要写了自定义 LambdaQueryWrapper / XML（如 `eq("status", 1)` 拼条件、`@Select` 复杂 join），仍属于**集成测试范围**（§3），而非"不必测"。MyBatis-Plus 把 `eq` 写成 `ne` 编译通过、ORM 不报错，但跑出错数据。
> - 真正"不必测"的只有：**没有手写 SQL** 的 `IService` 默认方法（`save`、`getById`、`listAll`、`removeById`）。

### 1.4 当前测试金字塔的实际形状 vs 应有形状

```
应有形状（健康，底厚顶薄）：
       │ 业务验收 │           ← 少量人工
      │  手工冒烟  │          ← 少量
     │    E2E      │         ← 中等
    │  集成测试      │       ← 中等
   │   单元测试        │     ← 大量（底座）

PSA 现状（倒金字塔风险，承重在顶部）：
       │ 业务验收 │           ← 有，靠 Admin 记忆
      │  手工冒烟  │          ← 有，主要防线
     │   E2E       │         ← 形同虚设（1 条 home 标题）
    │  集成测试      │       ← 🚧 1 个 spec，8 用例（login-action）
       │ 单测 │              ← 后端 fst-biz 5 个；PC 0 个；其他模块 0
```

**承重墙在金字塔上层（人工）而非底层（自动化）**。这是不健康结构。本文档第 11–16 章给出补齐方案。

### 1.5 测试人员视角的三个工作准则（贯穿全文）

任何新增测试用例，必须按以下三栏组织（与 PSA 测试人员协作约定一致）：

1. **功能层**：接口契约、参数校验、返回值、异常分支、状态变迁、并发/事务边界
2. **业务层**：真实业务场景下的语义正确性（审批流转是否符合业务规则、金额计算是否符合财务口径、状态机是否符合业务流程预期）
3. **补充视角（边界与隐性场景）**：从代码注释和需求中无法直接推导、但作为测试人员应当主动质疑的场景。详见 §13"业务边界场景测试模板"

**判断标准**：一份 PR 的测试用例若只覆盖 happy path + 一两个明显异常，属于不合格，应退回补充第 3 栏。

## 2. 后端单元测试体系

> **本章状态**：✅ 已落地（fst-biz 模块 5 个 P0 测试 + CI 卡门禁）
> **当前缺口**：覆盖率基线未量化、bpm-biz 历史测试债未治理、@Spy 模式被滥用、`MockedStatic` 忘记 close 偶有发生

### 2.1 框架已就绪（无需额外安装）

`yudao-spring-boot-starter-test` 已被所有业务模块依赖，包含以下所有组件，**不需要在 pom.xml 里单独添加**：

| 组件 | 用途 |
|---|---|
| **JUnit 5** (`junit-jupiter`) | 测试框架，`@Test` / `@DisplayName` / `@BeforeEach` 等注解 |
| **AssertJ** | 流式断言库，`assertThat` / `assertThatThrownBy` / `assertThatNoException` |
| **Mockito Inline** (`mockito-inline`) | 支持 mock `final` 类和 `static` 方法（`MockedStatic`），是普通 `mockito-core` 的超集 |
| **Spring Boot Test** | `@SpringBootTest` / `@MockBean`，集成测试时使用，单元测试不需要 |
| **H2** | 内嵌内存数据库，供集成测试使用，单元测试不需要 |
| **jedis-mock** (`fppt/jedis-mock`) | 内嵌内存 Redis，供集成测试使用 |
| **PODAM** | 随机生成 POJO 测试夹具，适合快速填充大 DO 对象 |

单元测试只需要 JUnit 5 + AssertJ + Mockito Inline，不启动 Spring 容器，无需 H2/jedis-mock。

### 2.2 测试文件存放规则

```
psa-backend/
└── yudao-module-fst/yudao-module-fst-biz/
    └── src/
        ├── test/java/cn/iocoder/yudao/module/fst/
        │   ├── service/manhour/
        │   │   ├── TimeSheet75HourRuleTest.java          ← 一条业务规则一个文件（推荐）
        │   │   ├── TimeSheetCancelEditableTest.java
        │   │   └── TimeSheetRateResolveStrictTest.java
        │   └── framework/aop/
        │       └── ParamOutAspectThreadLocalTest.java
        └── test-integration/java/                        ← 集成测试（详见第 3 节）
```

**命名规则（两种，按场景选）**：

1. **按类命名**：`被测类名 + Test.java`
   - 适用：一个类的多个方法都要测
   - 示例：`ManHourServiceImplTest.java`

2. **按业务规则命名**（推荐，现有代码采用此方式）：`{Domain}{规则关键词}Test.java`
   - 适用：守护一条具体规则，文件名即文档
   - 示例：`TimeSheet75HourRuleTest.java`、`TimeSheetCancelEditableTest.java`
   - 优点：CI 失败时一眼看出是哪条规则被破坏

**包路径约定**：测试类与被测类同包（`cn.iocoder.yudao.module.fst.service.manhour`），这样可以访问包级可见成员，无需全部改成 `public`。

### 2.4 常用断言速查

#### 异常断言

```java

        .isInstanceOf(ServiceException.class)
        .hasMessageContaining("不得低于7.5小时");

        .hasMessageContaining("档案字段不全")

        .hasMessageContaining("salaryGradeNo");

// 验证不抛任何异常（通过路径）
assertThatNoException()
        .isThrownBy(() -> service.submit(req));
```

#### 返回值断言

```java
// BigDecimal 比较（推荐用 isEqualByComparingTo，忽略精度差异：800 == 800.000）
assertThat(result.getStandardCostAmount()).isEqualByComparingTo("800.000");

// 普通对象
assertThat(result).isNotNull();
assertThat(result.getJobLevelCode()).isEqualTo("P3");

// 集合
assertThat(list).hasSize(3);

```

#### Mock 交互验证

```java
// 验证方法被调用了 1 次

// 验证方法从未被调用
verify(leaveRecordsMonthService, never()).getLeaveMinutes(any(), any());
```

#### doReturn vs when...thenReturn 的选择

```java
// @Spy 对象：必须用 doReturn，否则会真实执行被 mock 的方法（然后再替换返回值）
// 错误（会真实执行 appSubmitManHour，可能导致 NPE）：
when(manHourService.appSubmitManHour(any())).thenReturn(99L);

// 正确（直接跳过真实执行）：
doReturn(99L).when(manHourService).appSubmitManHour(any());

// @Mock 对象：两种写法等价，when...thenReturn 可读性更好
when(calendarService.getCalendarMap()).thenReturn(map);
doReturn(map).when(calendarService).getCalendarMap();
```

```java
// 当 stub 不是每个测试都会用到时（StrictStubs 模式下，未使用的 stub 会报错）
// 用 lenient() 标记为"宽松"：允许该 stub 未被某些测试调用
lenient().when(stringRedisTemplate.hasKey(any())).thenReturn(false);

// MockedStatic 里通常也需要 lenient，因为 PASS 路径和 REJECT 路径走到的静态调用不同
```

### 2.7 BigDecimal 断言强制规则

金额、工时、费率等 BigDecimal 字段断言**严禁使用 `equals` 或 `isEqualTo`**：

```java
// ❌ 错误：800 与 800.000 在 equals 下不相等（scale 不同，BigDecimal.equals 比较 scale）
assertThat(amount).isEqualTo(new BigDecimal("800"));   // 实际值是 800.000 会失败

// ✅ 正确：用 isEqualByComparingTo，仅比较数值忽略 scale
assertThat(amount).isEqualByComparingTo("800");
assertThat(amount).isEqualByComparingTo(new BigDecimal("800.000"));
```

**为什么这是高发坑**：业务代码中 BigDecimal 的 scale 经常被无意改变（`setScale` / `divide` 默认 scale=10），测试用 `isEqualTo` 在重构后可能突然红，定位耗时长且根因不明显。

**建议的自动化拦截**：在 `yudao-spring-boot-starter-test` 内加一条 ArchUnit 规则（详见 §9.5）禁止 `assertThat(BigDecimal).isEqualTo(...)`，强制走 `isEqualByComparingTo`。

### 2.8 推荐基类：AbstractStaticMockTest（消灭忘记 close 的 Bug）

`MockedStatic` 忘记 `close()` 会污染其他测试，最难排查（症状是"单独跑这个测试 PASS，全套跑就 FAIL"）。建议在 fst-biz 测试包下新建：

```java
package cn.iocoder.yudao.module.fst.test;

import org.junit.jupiter.api.AfterEach;
import org.mockito.MockedStatic;
import java.util.ArrayList;
import java.util.List;

/**
 * 静态 mock 测试基类：自动管理 MockedStatic 生命周期，避免泄漏污染其他测试。
 *
 * 用法：
 *   class YourTest extends AbstractStaticMockTest {
 *       private MockedStatic<SecurityFrameworkUtils> securityStatic;
 *
 *       @BeforeEach
 *       void setUp() {
 *           securityStatic = registerStatic(mockStatic(SecurityFrameworkUtils.class));
 *           // 不再需要 @AfterEach 手动 close
 *       }
 *   }
 */
public abstract class AbstractStaticMockTest {

    private final List<MockedStatic<?>> registered = new ArrayList<>();

    protected <T> MockedStatic<T> registerStatic(MockedStatic<T> mock) {
        registered.add(mock);
        return mock;
    }

    @AfterEach
    void closeRegisteredStaticMocks() {
        // 倒序关闭：模拟 try-with-resources 行为
        for (int i = registered.size() - 1; i >= 0; i--) {
            try {
                registered.get(i).close();
            } catch (Exception ignored) {
                // 已被手动 close 的情况下，忽略二次 close 异常
            }
        }
        registered.clear();
    }
}
```

**适用判断**：
- 单个 `MockedStatic` 仍可手动 close（保持与现有测试的一致性）
- **2 个以上必须用基类**（出错概率随数量指数上升）
- 涉及子类继承 `@SpringBootTest` 基类时，可改用 composition 而不是 inheritance（用 `@RegisterExtension`）

### 3.3 FST 模块集成测试基类（新建步骤）

当你需要给 FST 模块写集成测试（验证 Mapper SQL、Redis 操作等），步骤如下：

**Step 1**：在 `src/test-integration/java/cn/iocoder/yudao/module/fst/test/` 创建：

```java
public abstract class BaseFstIntegrationTest extends BaseDbAndRedisIntegrationTest {
    // 可在此处添加 FST 模块专用的测试工具方法
    // 例如：创建测试员工、初始化项目数据等

```

**Step 2**：确认 `src/test-integration/resources/application-integration-test.yaml` 存在（FST 模块已有，路径见上一节）。

**Step 3**：写集成测试类：

```java
package cn.iocoder.yudao.module.fst.dal.mysql.manhour;

import cn.iocoder.yudao.module.fst.test.BaseFstIntegrationTest;
import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.transaction.annotation.Transactional;

import static org.assertj.core.api.Assertions.assertThat;

@Transactional   // 每个测试自动回滚，测试间互不污染
class ManHourMapperIntegrationTest extends BaseFstIntegrationTest {

    @Autowired
    private ManHourMapper manHourMapper;

    @Test
    void selectByPersonAndDate_returnsCorrectRecord() {
        // given：在 H2 内存库里插入测试数据
        ManHourDO record = new ManHourDO();
        // ... 填充字段 ...
        manHourMapper.insert(record);

        // when
        ManHourDO found = manHourMapper.selectOne(
            ManHourMapper::getPersonId, record.getPersonId());

        // then
        assertThat(found).isNotNull();
        assertThat(found.getDeclareHours()).isEqualByComparingTo("8.0");
    }
}
```

**Step 4**：运行集成测试（不加入 CI，只在本地跑）。

> ⚠️ **重要前置**：surefire 默认 includes 模式是 `**/Test*.java, **/*Test.java, **/*Tests.java, **/*TestCase.java`，**会排除 `*IntegrationTest.java`**。直接 `mvn test` 跑出来是 0 测试。文档之前给的 `-Dtest="*IntegrationTest"` 命令也只能识别 surefire 已 include 的类，配合 includes 配置才有效。

**方案 A：复用 surefire（最少配置改动）**

在 `psa-backend/yudao-module-fst/yudao-module-fst-biz/pom.xml` 添加 surefire `<includes>`，并把 test-integration 源目录加入 build-helper：

```xml
<build>
  <plugins>
    <plugin>
      <groupId>org.codehaus.mojo</groupId>
      <artifactId>build-helper-maven-plugin</artifactId>
      <executions>
        <execution>
          <id>add-integration-test-source</id>
          <phase>generate-test-sources</phase>
          <goals><goal>add-test-source</goal></goals>
          <configuration>
            <sources>
              <source>src/test-integration/java</source>
            </sources>
          </configuration>
        </execution>
        <execution>
          <id>add-integration-test-resources</id>
          <phase>generate-test-resources</phase>
          <goals><goal>add-test-resource</goal></goals>
          <configuration>
            <resources>
              <resource><directory>src/test-integration/resources</directory></resource>
            </resources>
          </configuration>
        </execution>
      </executions>
    </plugin>
    <plugin>
      <artifactId>maven-surefire-plugin</artifactId>
      <configuration>
        <includes>
          <include>**/*Test.java</include>
          <include>**/*IntegrationTest.java</include>   <!-- 显式纳入 -->
        </includes>
      </configuration>
    </plugin>
  </plugins>
</build>
```

跑命令：

```bash
cd .worktrees/issue-N/psa-backend
mvn -B -ntp -DskipTests -pl yudao-module-fst/yudao-module-fst-biz -am install
mvn -B -ntp -pl yudao-module-fst/yudao-module-fst-biz \
    -Dtest="*IntegrationTest" test
```

**方案 B：用 failsafe（更标准，单测/集成测严格分离）—— 推荐**

```xml
<plugin>
  <artifactId>maven-failsafe-plugin</artifactId>
  <version>3.0.0-M5</version>
  <executions>
    <execution>
      <goals>
        <goal>integration-test</goal>
        <goal>verify</goal>
      </goals>
    </execution>
  </executions>
  <configuration>
    <includes>
      <include>**/*IntegrationTest.java</include>
      <include>**/*IT.java</include>
    </includes>
  </configuration>
</plugin>
```

跑命令：

```bash
# 单测仍用 mvn test，集成测试用 mvn verify
mvn -B -ntp -pl yudao-module-fst/yudao-module-fst-biz verify
```

**两方案对比**：

| 维度 | 方案 A（surefire 扩 include） | 方案 B（failsafe） |
|---|---|---|
| 配置改动 | 小 | 中 |
| 与现有 `mvn test` 习惯 | 一致 | 需切到 `mvn verify` |
| 单测 / 集成测分离 | 弱（都走 surefire） | 强（surefire + failsafe 各管一半） |
| CI 分阶段跑 | 难 | 易（test 阶段快、verify 阶段慢可独立调度）|
| Maven 官方推荐 | — | ✅ |

**推荐方案 B**，为未来"CI 把集成测试独立挪到 nightly run"留路。

### 3.4 H2 模拟 SQL Server 的盲区（必读）

> **当前默认用 H2 跑集成测试，但 PSA prod 是 SQL Server**。两者并非完全兼容，"集成测试通过 ≠ prod 通过"。

#### 已知差异清单

| 差异点 | H2 (MODE=MySQL) | SQL Server | 影响 |
|---|---|---|---|
| 分页语法 | `LIMIT n OFFSET m` | `OFFSET n ROWS FETCH NEXT m ROWS ONLY` 或 `TOP n` | MyBatis-Plus 分页器会自适应，但**手写 SQL** 时会暴雷 |
| `MERGE` / `UPSERT` | 支持但语法不同 | `MERGE INTO ... USING ... WHEN MATCHED THEN ...` | 工时合并写入逻辑 |
| 临时表 | `CREATE LOCAL TEMPORARY TABLE` | `#tmp_xxx` | 报表存储过程 |
| 函数名 | `IFNULL`、`CONCAT` | `ISNULL`、`+` 拼接 | DTO 投影查询 |
| 大小写敏感 | 默认大小写不敏感（MySQL 模式） | PSA prod 是 `Chinese_PRC_CI_AS`（不敏感），但表名 / 字段名在某些设置下敏感 | 字段名拼写错误 |
| 中文排序 | UTF-8 字典序 | Chinese_PRC_CI_AS（拼音） | 列表排序结果 |
| `DATETIME` 精度 | 毫秒 | `DATETIME2(7)` 100 纳秒 | 时间字段比对 |
| 触发器 / 存储过程 | 几乎不支持 | 全支持 | PSA 有少量触发器逻辑 |
| 锁机制 | 行锁、表锁简化 | 行锁、表锁、键锁、范围锁 | 并发场景 |
| 自增主键回填 | `IDENTITY` 通用 | `OUTPUT INSERTED.id` 或 `SCOPE_IDENTITY()` | MyBatis-Plus 已封装，手写 SQL 会踩 |
| `nvarchar` vs `varchar` | 不区分 | 区分（中文必须 N 前缀）| 直接拼 SQL 时容易乱码 |

#### 应对策略（按强度排序）

**策略 1：明确"H2 集成测试只测 ORM 映射 + Mapper 逻辑"**

- ✅ 适合：`@TableField` 注解正确性、LambdaQueryWrapper 条件拼接、DTO 投影字段映射、关联查询、Redis 操作
- ❌ 不适合：自定义 XML SQL（含数据库方言）、存储过程调用、复杂 join 性能

**策略 2：Testcontainers 启动真实 SQL Server（推荐用于命脉模块）**

```xml
<dependency>
  <groupId>org.testcontainers</groupId>
  <artifactId>mssqlserver</artifactId>
  <version>1.19.7</version>
  <scope>test</scope>
</dependency>
```

**收益**：消除 H2/prod 差异盲区，SQL 兼容性 100% 被验证。
**代价**：单测启动 30-60s（H2 仅 3-5s），不适合每次 commit 跑，**仅本地按需 + dev 部署前周期性跑**。
**前置**：本地需装 Docker；CI runner 也需 Docker-in-Docker 支持（self-hosted runner 已具备）。

**策略 3：手写 SQL 必须在 dev 环境真实跑过**

如果业务紧迫无法等 Testcontainers 落地，强制约定：

- 凡是修改 `*Mapper.xml` 或包含 `@Select` / `@Update` 注解的 PR
- Test Plan **必须**包含一条 `- [x] dev 环境 SQL 真实执行验证通过`
- 否则 PR Review 退回

**当前 PSA 默认采用策略 1（保守）+ 策略 3（强制）**。命脉模块（工时审批、月度结算、费率计算）建议升级到策略 2。

```js
/**
 * 业务背景：
 *   parseTime 是 PSA 全局时间格式化函数，工时申报、审批时间、报表等
 *   超过 200 处调用点依赖它。一次格式错误会导致全站时间显示错乱。
 *
 * 功能层验证：
 *   - 入参类型：Date / number / string / null / undefined
 *   - format 参数：默认 {y}-{m}-{d} {h}:{i}:{s}、自定义、星期表达式 {a}
 *
 * 业务层验证：
 *   - 工时申报常用 worksDate 时间字符串能正确解析
 *   - 审批时间显示符合财务口径
 *
 * 补充视角（边界）：
 *   - 月日补零（2026-1-5 → 2026-01-05）
 *   - 跨时区（UTC 字符串 vs 本地时间）
 *   - 闰秒 / 非法日期（2026-02-30）容错
 */
import { parseTime, addDateRange, handleTree } from '@/utils/ruoyi'

describe('parseTime — 时间格式化', () => {

  describe('功能层', () => {
    test('Date 对象按默认格式 → YYYY-MM-DD HH:mm:ss', () => {
      const date = new Date(2026, 4, 8, 14, 30, 45)  // 用本地时间构造避免时区干扰
      expect(parseTime(date)).toBe('2026-05-08 14:30:45')
    })

    test('format = {y}/{m}/{d} → 自定义分隔符', () => {
      const date = new Date(2026, 4, 8, 0, 0, 0)
      expect(parseTime(date, '{y}/{m}/{d}')).toBe('2026/05/08')
    })

    test.each([null, undefined, ''])('入参 %p → 返回 null', (input) => {
      expect(parseTime(input)).toBeNull()
    })

    test('Unix 时间戳（数字）也能解析', () => {
      const ts = new Date(2026, 4, 8, 12, 0, 0).getTime()
      expect(parseTime(ts)).toMatch(/^2026-05-08 12:00:00$/)
    })
  })

  describe('业务层', () => {
    test('工时 worksDate 字符串 "2026-05-08 12:00:00" 应原样回显', () => {
      // worksDate 在后端是 LocalDateTime，前端展示需要稳定
      const result = parseTime('2026-05-08 12:00:00')
      expect(result).toBe('2026-05-08 12:00:00')
    })
  })

  describe('补充视角（边界）', () => {
    test('个位月份必须补零（防止表格列宽抖动）', () => {
      const date = new Date(2026, 0, 5)  // 1月5日
      expect(parseTime(date, '{y}-{m}-{d}')).toBe('2026-01-05')
    })

    test('闰年 2 月 29 日不应被识别为 3 月 1 日', () => {
      const leap = new Date(2024, 1, 29)  // 2024 是闰年
      expect(parseTime(leap, '{y}-{m}-{d}')).toBe('2024-02-29')
    })

    test('非法字符串应容错返回 null 或原值，不抛异常', () => {
      expect(() => parseTime('not-a-date')).not.toThrow()
    })
  })
})

  ]

  describe('功能层', () => {
    test('正常 4 节点 → 1 个根 + 2 子 + 1 孙', () => {
      const tree = handleTree(flat)
      expect(tree).toHaveLength(1)
      expect(tree[0].children).toHaveLength(2)
      expect(tree[0].children[0].children).toHaveLength(1)
    })

    test('空数组 → 空数组（不抛异常）', () => {
      expect(handleTree([])).toEqual([])
    })
  })

  describe('补充视角（边界）', () => {
    test('循环引用（A→B→A）→ 不死循环（源码是扁平 O(n²) filter，无递归）', () => {
      const cycle = [
        { id: 1, parentId: 2 },
        { id: 2, parentId: 1 }
      ]
      // 源码用 cloneData.filter 二层嵌套，但没有 children 递归，因此不会死循环
      // rootId = Math.min(abs(2), abs(1)) = 1，所以 id=2 被识别为"根"
      const result = handleTree(cycle)
      expect(result).toHaveLength(1)
      expect(result[0].id).toBe(2)
      // id=2 下面挂着 id=1 作为 children（即使 id=1 本身也是 id=2 的 parent）
      expect(result[0].children).toHaveLength(1)
      expect(result[0].children[0].id).toBe(1)
    })

    test('孤儿节点（parentId 指向不存在的 id）→ 不出现在树中', () => {
      const orphan = [
        { id: 1, parentId: 0 },
        { id: 99, parentId: 88 }  // 88 不存在
      ]
      // rootId 默认计算：min(abs(0), abs(88)) = 0
      const tree = handleTree(orphan)
      expect(tree).toHaveLength(1)
      expect(tree[0].id).toBe(1)
      // id=99 因 parentId !== rootId 被 filter 过滤掉，不出现
    })

    test('自定义 rootId / id / parentId 字段名', () => {
      const flat = [
        { code: 'A', pcode: 'ROOT', label: '根' },
        { code: 'B', pcode: 'A', label: '子' }
      ]
      const tree = handleTree(flat, 'code', 'pcode', 'kids', 'ROOT')
      expect(tree).toHaveLength(1)
      expect(tree[0].code).toBe('A')
      // 注意：源码用的字段名仍然是 .children 而非 kids（参数 children 实际未生效，是源码 Bug）
      // 此测试锁定当前行为；如果某天源码修复了，这条测试会红，提醒升级文档
      expect(tree[0].children).toBeDefined()
```

### 4.6 测试示例（Vuex store/permission.js mutations）

`tests/unit/store/permission.spec.js`（新建）：

```js
/**
 * 业务背景：
 *   permission store 是路由权限注入的核心。SET_ROUTES 错误会导致整个
 *   左侧菜单消失（权限被吞）或暴露未授权菜单（越权可见）。
 *
 * 功能层：mutations 是纯函数，单测覆盖所有 SET_* 的状态变更
 * 业务层：合并 constantRoutes 顺序正确、addRoutes 追加在尾部
 * 补充视角：
 *   - 重复调用 SET_ROUTES 不应累加（防止历史路由残留）
 *   - 路由对象 children 嵌套深度 5+ 仍正确
 */
import permissionModule from '@/store/modules/permission'

const { mutations } = permissionModule

describe('permission store mutations', () => {

  let state

  beforeEach(() => {
    state = {
      routes: [],
      addRoutes: [],
      sidebarRouters: [],
      topbarRouters: []
    }
  })

  describe('SET_ROUTES — 功能层', () => {
    test('设置 addRoutes 并合并 constantRoutes', () => {
      const newRoutes = [{ path: '/project', name: 'Project' }]
      mutations.SET_ROUTES(state, newRoutes)

      expect(state.addRoutes).toEqual(newRoutes)
      expect(state.routes.length).toBeGreaterThan(newRoutes.length)
      expect(state.routes).toContainEqual(newRoutes[0])
    })
  })

  describe('SET_ROUTES — 补充视角', () => {
    test('重复调用不应累加（防止历史路由残留）', () => {
      const r1 = [{ path: '/a' }]
      const r2 = [{ path: '/b' }]
      mutations.SET_ROUTES(state, r1)
      const len1 = state.routes.length
      mutations.SET_ROUTES(state, r2)
      const len2 = state.routes.length

      // 第二次调用后，长度应该是 constantRoutes + r2，不是 constantRoutes + r1 + r2
      expect(len2).toBe(len1)
      expect(state.routes).toContainEqual(r2[0])
      expect(state.routes).not.toContainEqual(r1[0])
    })

    test('入参 [] → addRoutes 为 []，routes 仍含 constantRoutes', () => {
      mutations.SET_ROUTES(state, [])
      expect(state.addRoutes).toEqual([])
      expect(state.routes.length).toBeGreaterThan(0)
    })
  })

  describe('SET_SIDEBAR_ROUTERS', () => {
    test('更新 sidebarRouters', () => {
      const routes = [{ path: '/bpm', name: 'Bpm' }]
      mutations.SET_SIDEBAR_ROUTERS(state, routes)
      expect(state.sidebarRouters).toEqual(routes)
    })
  })
```

### 4.9.0 集成测试的双重价值（先读，再决定要不要写）

很多人以为集成测试的价值是 "**自动逮回归 Bug**"。这只是它的**下半场**。它的**上半场**价值更值钱：

> **集成测试强迫开发者用测试人员视角逐项枚举边界场景，**
> **倒逼细读源码，让本来潜伏的 Bug 在写测试的当天就被发现。**

| 时间维度 | 价值类型 | 是谁在工作 |
|---|---|---|
| **写 spec 前** —— "上半场" | **思维强制**：被迫枚举功能/业务/边界三栏，被迫读源码理解契约 | **写测试这件事本身** |
| **spec 跑通后** —— "下半场" | **防回归**：未来谁改坏了源码，CI 立刻红 | **测试代码** |

**首条 spec 的实证**：写 `login-action.spec.js` 的"补充视角"栏时，被准则强制枚举边界 → 想到"后端只返 accessToken 不返 refreshToken 会怎样" → 读 `src/utils/auth.js#setToken` → 发现没判空、`undefined` 会被持久化成字符串 `"undefined"`。这个 Bug 在 PSA 已存在多个版本，没被任何自动化扫描或回归测试逮到，**是写集成测试当天被发现并修复的**。

**对团队的实用结论**：
- 不要把集成测试当成"防回归的开销"，它的首次写作收益本身就足够回本
- PR 描述写"集成测试发现 Bug"是过誉，但写"撰写集成测试边界用例时通过 code reading 发现 Bug"是诚实且有价值的记录
- "测试人员视角的三栏组织（功能 / 业务 / 补充视角）" 不是形式主义，是**思维强制装置**，省略=放弃上半场价值

### 4.9.1 三层测试的分工口诀

| 层 | 关心什么 | mock 什么 | 速度 | 例 |
|---|---|---|---|---|
| **单元（§4）** | 单个函数 / mutation 的逻辑 | 一切外部 | < 100ms | `parseTime('2026-05-08')` 返回值正确 |
| **集成（本节）** | 多模块协作正确 | 只 mock 网络（axios） | < 1s | `Login` action 拿到 token 后 `setToken` 写入 localStorage |
| **E2E（§11）** | 真实业务路径，包括后端 | 不 mock | > 10s | 真实账号登录 → 跳转 dashboard |

判断口诀：**纯函数 → 单元；多文件协作但不需要后端 → 集成；要真后端 → E2E**。

### 4.9.5 关键脚手架细节

#### 切断重型 cascade（setup.js 必须）

PSA 的 `src/utils/request.js` 顶层 `import store from '@/store'`，而 store/index.js 进而 import permission → router/index.js → `new Router({...})` 含 webpack-only 的 lazy-load 字符串 component，在 Jest 下崩。

**解法**：setup.js 顶部 `jest.mock('@/store', ...)` 和 `jest.mock('@/router', ...)`：

```js
jest.mock('@/store', () => ({
  __esModule: true,
  default: { dispatch: jest.fn(() => Promise.resolve()), state: {}, getters: {} }
}))

jest.mock('@/router', () => ({
  __esModule: true,
  default: { push: jest.fn(() => Promise.resolve()), replace: jest.fn(),
             currentRoute: { path: '/', query: {} } }
}))
```

这样 request.js 能加载、interceptor 完整跑，但不拉整个 app cascade。

#### 拦截"业务 axios 实例"而非顶层 axios

PSA 的 service 是 `axios.create({ baseURL: '/admin-api/' })`。`axios-mock-adapter` 默认拦截顶层 `axios`，**对 service 实例无效**。`mountWithDeps.js` 提供 `mockBusinessAxios()`：

```js
export function mockBusinessAxios() {
  const service = require('@/utils/request').default
  return new MockAdapter(service)
}
```

#### moduleNameMapper 顺序敏感

```js
// ❌ 错误：宽松规则在前会吃掉 @/layout 的精确匹配
moduleNameMapper: {
  '^@/(.*)$': '<rootDir>/src/$1',
  '^@/layout$': '<rootDir>/tests/unit/__mocks__/fileMock.js',  // 永远不会命中
}

// ✅ 正确：精确规则在前
moduleNameMapper: {
  '^@/layout$': '<rootDir>/tests/unit/__mocks__/fileMock.js',
  '^@/layout/(.*)$': '<rootDir>/tests/unit/__mocks__/fileMock.js',
  '^@/(.*)$': '<rootDir>/src/$1'   // 兜底放最后
}
```

| 层 | 用例 | 验证 |
|---|---|---|
| 功能层 | 成功登录 → POST 到 /system/auth/login，payload 含 username/password | URL + method + payload 字段 |

| 业务层 | 成功登录 → ACCESS_TOKEN + REFRESH_TOKEN 都被持久化 | setToken 行为 |
| 业务层 | 后端 HTTP 500 → action reject，localStorage 无 token | 错误路径 |

> **发现机制（精确措辞）**：**不是**测试运行时自动逮到的，而是**写 spec 的补充视角栏时，被边界枚举准则强制问"如果只返一个字段会怎样"，进而读 `setToken` 源码发现的**。这是 §4.9.0 所说"集成测试的上半场价值（思维强制）"的活样本。
>
> **后续防回归**：本 spec 同步锁定修复后的行为（`toBeNull()`），未来谁不慎让 setToken 倒退回不判空逻辑，CI 立刻红 —— 这是"下半场价值（防回归）"接力上场。

## 9. 常见 Bug 预防规则

### 9.1 后端

| 问题 | 根因 | 预防方法 |
|---|---|---|
| 业务状态被绕过 | 入口 guard 未做单测，被遗漏 | 每个状态守卫方法必须有单测；P0 清单只增不减 |
| BigDecimal 精度丢失 | 用 `double` 或 `float` 参与金额计算 | 金额字段全程用 `BigDecimal`；测试用 `isEqualByComparingTo` 断言，不用 `equals` |
| 并发重复写入 | 无幂等设计 | DB 唯一索引 + 接口幂等（Redis 防重令牌）|
| BPM 与业务表状态不同步 | 直接改业务表而不走 BPM 事件回调 | 业务状态变更必须由 BPM Listener 驱动 |
| ThreadLocal 在池化线程泄漏 | 异步场景下未 `remove()`，导致下一个请求拿到上一个用户的数据 | 测试时用 `ParamOutAspectThreadLocalTest` 模式：`@BeforeEach` drain，`@AfterEach` drain |
| AOP 同类内调用失效 | Spring 代理边界问题：`this.method()` 不过代理 | 需要 AOP 生效的同类调用：注入自身或拆成单独 Bean；`@Spy` 单测时用 `doReturn` 可规避 |
| MockedStatic 污染其他测试 | 忘记在 `@AfterEach` 里 `close()` | 每个 `MockedStatic<>` 字段都在 `@AfterEach` 里显式关闭 |
| StrictStubs 误报 | stub 了但未被某个测试使用 | 用 `lenient()` 标记"不是每个测试都会用到"的 stub |

### 9.2 PC 前端

| 问题 | 预防方法 |
|---|---|
| 懒加载路由未覆盖，组件/样式不加载 | 改 `routeHeavyDeps.js` 后：用 grep 验证所有路径段已在 `BPMN_DESIGNER_PATH_SEGMENTS` 和 `VXE_PATH_SEGMENTS` 中覆盖 |
| BPMN 渲染竞争（快速切路由）| 用 `renderVersion` 计数器 + Promise chain 串行化；`ProcessViewer.vue` 中已有此模式，勿删除 |
| echarts 实例未销毁内存泄漏 | `beforeDestroy` 生命周期里 `chart.dispose()`；改 chart 组件时检查是否有 dispose 逻辑 |
| Vuex action 错误被静默吞掉 | async action 调用方必须 `await` + `.catch()`；组件里不要裸调 `this.$store.dispatch()` 忘记处理 rejected |
| `require.context` 找不到视图 | component 字符串 → 视图文件的映射，按三候选路径顺序：`./x.vue` → `./x/index.vue` → `./x/Index.vue` |
| babel 环境配置：`dynamic-import-node` 与 `jest.mock()` 潜在冲突 | 当前 `babel.config.js` 的 test env 保留 `dynamic-import-node`（路由懒加载需要）。如果某条 spec 出现 `jest.mock('@/api/xxx')` 不生效，临时改用 `jest.doMock('@/api/xxx')`；详见 §4.2 注释 |
| 路由跳转类型漂移 | 路由 `params` 里的数字字段（如 ID）用 `String()` 或 `:prop="String(id)"` 传参，不要混用 `number` 和 `string` |

### 9.3 小程序（psa-mini-program-new）

| 问题 | 预防方法 |
|---|---|
| 联调打到他人机器 | 只通过 `serve-mini-worktree.sh` 启动，禁止在 `config.js` 硬编码 IP；每次提交前 `git diff config.js` 确认没有残留本机 IP |
| token 失效未跳转登录 | `request.ts` 统一拦截 401 → `reLaunch qyLogin`；测试时故意清空 token 验证跳转 |
| 页面数据不刷新（回退后旧数据） | `onShow` 做数据刷新，不只依赖 `onLoad`；列表页 + 详情页都要验证"返回后数据是否更新" |
| 重新发起工时 originId 缺失 | 参见 #157/#158：发起前检查 originId 字段是否正确带到请求体 |
| 组织字段 orgXxx 缺失 | 参见 #157：`submitProcessEditManHour` 必须带 3 个 org 字段，worktree 里通过抓包验证请求体 |

### 9.4 psa-mini（Taro + React）

| 问题 | 预防方法 |
|---|---|
| Taro API 在 jsdom 中直接调用崩溃 | 所有 `@tarojs/taro` 的调用必须通过 `tests/unit/__mocks__/taro.ts` mock，不在测试里 import 真实 Taro |
| zustand store 在测试间相互污染 | 每个测试 `beforeEach` 里 `useXxxStore.setState(initialState)` 重置；或用 `jest.isolateModules()` |
| ts-jest 编译 tsx 时 jsx 配置错误 | `jest.config.ts` 中 ts-jest 的 `tsconfig.jsx` 必须为 `'react'`（不是 `'preserve'`），通过 `globals` 配置 |
| CSS Modules 类名在测试中变成 `[object Object]` | 确认 `moduleNameMapper` 中有 `identity-obj-proxy` 覆盖 `.css` / `.scss` |

### 9.5 把"约定"升级为"自动化拦截"

§9.1–9.4 列的预防规则全部是**口头约定 + Code Review 兜底**。下一个新人不知道、Reviewer 当天疲劳，就会漏。
**根治办法**是把约定写成机器可执行的规则。

#### 后端：ArchUnit 规则

`yudao-spring-boot-starter-test` 添加 `archunit-junit5` 依赖，在 fst-biz 测试包下新建 `architecture/ArchitectureRulesTest.java`：

```xml
<dependency>
  <groupId>com.tngtech.archunit</groupId>
  <artifactId>archunit-junit5</artifactId>
  <version>1.2.1</version>
  <scope>test</scope>
</dependency>
```

```java
package cn.iocoder.yudao.module.fst.architecture;

import com.tngtech.archunit.junit.AnalyzeClasses;
import com.tngtech.archunit.junit.ArchTest;
import com.tngtech.archunit.lang.ArchRule;

import java.math.BigDecimal;

import static com.tngtech.archunit.lang.syntax.ArchRuleDefinition.*;

@AnalyzeClasses(packages = "cn.iocoder.yudao.module.fst")
class ArchitectureRulesTest {

    // 规则 1：金额 / 工时 / 费率字段禁止用 double / float
    @ArchTest
    static final ArchRule moneyFieldsMustBeBigDecimal =
        noFields()
            .that().haveNameMatching(".*(Amount|Cost|Hours|Rate|Price).*")
            .should().haveRawType(double.class)
            .orShould().haveRawType(Double.class)
            .orShould().haveRawType(float.class)
            .orShould().haveRawType(Float.class)
            .as("金额 / 工时 / 费率字段必须用 BigDecimal");

    // 规则 2：BigDecimal 字段断言禁止用 isEqualTo（precision 陷阱）
    @ArchTest
    static final ArchRule bigDecimalAssertionMustUseCompareTo =
        noClasses()
            .that().resideInAPackage("..test..")
            .should().callMethod(org.assertj.core.api.AbstractBigDecimalAssert.class, "isEqualTo", Object.class)
            .as("BigDecimal 断言用 isEqualByComparingTo，不要用 isEqualTo");

    // 规则 3：Service 方法返回 BigDecimal 时不允许返回 null（业务侧约定返回 ZERO）
    // 这条 ArchUnit 难直接表达，建议用 SonarQube 自定义规则；此处仅作示意

    // 规则 4：测试代码禁止用 LocalDateTime.of(*,*,*,0,0) 容易踩跨时区
    @ArchTest
    static final ArchRule testCodeMustNotUseZeroHourTime =
        noClasses()
            .that().resideInAPackage("..test..")
            .should().callMethodWhere(target ->
                target.getOwner().isEquivalentTo(java.time.LocalDateTime.class)
                && target.getName().equals("of")
                // 这里实际需要 ArchUnit 自定义 condition 检查参数值
            )
            .as("测试构造时间用中午 12:00 避免跨时区，禁止 0:00");
    // 注：第 4 条用 ArchUnit 直接表达需要写自定义 Condition，复杂度较高
    //     更务实的方案是写一个 Checkstyle / SpotBugs 自定义规则
}
```

把这个 ArchitectureRulesTest 加入 fst-biz 的 surefire 必跑清单，破坏规则的 PR 在 CI 直接红。

**收益**：从此"BigDecimal 用 isEqualTo"、"金额字段用 double"等坑，PR 阶段就被拦下。

**渐进路径**：先加规则 1、2（最高价值，实现简单），后面再补 3、4。一次性把所有规则上线会因历史代码不合规导致 CI 全红，需要先做一次"清算迁移 PR"再加规则。

#### 后端：Checkstyle 补 ArchUnit 难表达的规则

某些规则 ArchUnit 不擅长（如"禁止 `Thread.sleep` 在测试代码"、"接口必须有 Javadoc"），用 Checkstyle 更合适。`yudao-spring-boot-starter-test` 加 maven-checkstyle-plugin，自定义规则集放在 `psa-backend/checkstyle/psa-rules.xml`。

#### PC 前端：自定义 lint 脚本（grep 级别）

部分约定写 ESLint 规则成本高，可以用 shell 脚本在 `pr-gate.yml` 跑：

```bash
# psa-front-desk-pc/scripts/check-conventions.sh
#!/bin/bash
set -e

# 检查 1：routeHeavyDeps.js 中的路径段是否在 src/router/ 下都能找到
node scripts/verify-route-heavy-deps.js

# 检查 2：禁止在 src/api/ 下硬编码 baseURL
if grep -RnE "http://(localhost|127|192|10|172)" src/api/; then
  echo "ERROR: src/api/ 不允许硬编码 IP / baseURL"
  exit 1
fi

# 检查 3：禁止在 src/ 下提交 console.log（开发临时调试遗留）
if grep -RnE "^\s*console\.log\(" src/ --include="*.js" --include="*.vue"; then
  echo "WARN: 检测到 console.log，请清理"
  # 不 exit 1，仅警告
fi
```

挂到 `frontend-changed-lint` job 末尾：

```yaml
- name: PC 前端约定检查
  working-directory: psa-front-desk-pc
  run: bash scripts/check-conventions.sh
```

#### 把规则升级路径写进 §16 跨模块影响清单

每条新加的自动化规则都会"挡住"一批历史代码。建议：

模板示例（`tests/e2e/login.spec.js`）：

```js
const { test, expect } = require('@playwright/test')

/**
 * 业务背景：
 *   登录是所有功能入口。一旦 token 写入或跳转链路出问题，整个系统不可用。
 *   E2E 不验证内部状态（那是单测），只验证"用户视角看到正确的页面变化"。
 *
 * 功能层：表单提交 + 跳转 + token 写入 localStorage
 * 业务层：登录成功后看到自己有权访问的菜单（按角色不同）
 * 边界视角：错误密码、token 过期跳转、网络异常
 */
test.describe('登录流程', () => {

  test('正确账号密码 → 跳转 Dashboard 并写入 token', async ({ page }) => {
    await page.goto('/login')

    await page.getByPlaceholder('账号').fill(process.env.E2E_USER || 'admin')
    await page.getByPlaceholder('密码').fill(process.env.E2E_PASS || 'admin123')
    await page.getByRole('button', { name: '登 录' }).click()

    // 业务层断言：跳转到首页
    await expect(page).toHaveURL(/\/(index|dashboard)/, { timeout: 10000 })

    // 功能层断言：token 写入 localStorage
    const token = await page.evaluate(() => localStorage.getItem('ACCESS_TOKEN'))
    expect(token).toBeTruthy()
  })

  test('错误密码 → 提示且停留在登录页', async ({ page }) => {
    await page.goto('/login')
    await page.getByPlaceholder('账号').fill('admin')
    await page.getByPlaceholder('密码').fill('wrong-password')
    await page.getByRole('button', { name: '登 录' }).click()

    // 业务层：错误提示出现
    await expect(page.getByText(/密码错误|账号或密码/)).toBeVisible()
    // 功能层：URL 没跳走
    await expect(page).toHaveURL(/\/login/)
  })
})
```

> ⚠️ **测试账号管理**：E2E 用账号必须**独立于真人账号**，建议在 dev 环境建 `e2e_test` 用户，密码用 GitHub Secret `E2E_USER` / `E2E_PASS` 注入。**严禁**用真实员工账号跑 E2E（误操作会污染审批流）。

#### 阶段 3（一个月内）：接入 CI（按需触发）

E2E 比单测慢一个量级（一次跑 3 条用例 ~30-60s + 前后端启动 ~60s），不适合每次 push 都跑。建议**按需触发**：

```yaml
# .github/workflows/pr-gate.yml 新增 job（独立，不进 pr-gate 汇聚必过）
e2e-pc:
  name: PC E2E (Playwright)
  runs-on: self-hosted
  if: contains(github.event.pull_request.labels.*.name, 'run-e2e') || github.ref == 'refs/heads/develop'
  # 仅当 PR 打上 run-e2e 标签 或 push 到 develop 时触发
  timeout-minutes: 20
  steps:
    - uses: actions/checkout@v4
    - uses: pnpm/action-setup@v4
      with: { version: 7.33.7 }
    - uses: actions/setup-node@v4
      with: { node-version: '16' }
    - name: Install dependencies
      working-directory: psa-front-desk-pc
      run: pnpm install --frozen-lockfile
    - name: Install Playwright browsers
      working-directory: psa-front-desk-pc
      run: pnpm exec playwright install chromium
    - name: Start backend & PC (via global-setup)
      working-directory: psa-front-desk-pc
      env:
        E2E_USER: ${{ secrets.E2E_USER }}
        E2E_PASS: ${{ secrets.E2E_PASS }}
      run: pnpm run test:e2e -- --project=chromium
    - name: Upload Playwright report
      if: always()
      uses: actions/upload-artifact@v4
      with:
        name: playwright-report
        path: psa-front-desk-pc/playwright-report
```

**为什么不进 pr-gate 必过**：E2E 偶发 flaky（网络、时序），如果硬卡门禁会把整个 PR 流程拖慢。当前阶段定位为"可选回归"，labe-driven 触发。等用例稳定 3 个月以上、Flaky 率 <2% 时再考虑硬卡。

### 11.3 编写 E2E 用例的纪律

| 纪律 | 说明 |
|---|---|
| **不写"全端验证"** | E2E 只验证用户视角看得见的关键状态变化，不验证内部状态、数据库行数等（那是单测/集成测试的活）|
| **每条用例独立** | 用 `test.beforeEach` 重置登录态、清理脏数据，不依赖前一条用例的副作用 |
| **不用 `nth-child`/`xpath` 抓元素** | 用 `getByRole`、`getByPlaceholder`、`getByText`，对 UI 重构鲁棒 |
| **断言用业务语义** | `expect(toast).toHaveText('提交成功')` 而不是 `expect(screenshot).toMatchSnapshot()`（截图断言极度 flaky）|
| **慎用 `waitForTimeout`** | 用 `waitForSelector` / `waitForResponse` 显式等条件，固定等待是 Flaky 之源 |
| **失败先看 trace** | `trace: 'retain-on-failure'` 已配，失败时 `playwright-report/` 里有完整时间线 |

#### 11.5.1 区别与定位

| | `@playwright/test`（§11.1–11.4） | `@playwright/mcp`（本节）|

`.mcp.json` 实际内容：

```json

    "playwright": {
      "command": "npx",
      "args": [
        "-y",
        "@playwright/mcp@latest",
        "--browser=chromium",
        "--viewport-size=1440,900",
```

**关键参数解释**：

| 参数 | 作用 | 为什么这么选 |
|---|---|---|
| `--browser=chromium` | 用 Playwright 自带 chromium | 与 `playwright.config.js` 一致；零外部依赖 |

| `--ignore-https-errors` | 忽略 SSL 错误 | dev 环境 / 自签证书容错 |

## 12. 覆盖率棘轮 CI 强制机制

> **本章状态**：🚧 jest.config.js 已配 `coverageThreshold` 基线为 0（§4.2），但 CI 跑的 `pnpm run test:unit` 不带 `--coverage`，棘轮当前是哲学**不是机制**。本章给出把"只增不减"变成 CI 卡门禁的方案。

### 12.1 现状与缺口

| 当前 | 应有 |
|---|---|
| `jest.config.js` 有 `coverageThreshold: { lines: 0, ... }` | ✅ 占位已就绪 |
| CI 跑 `pnpm run test:unit`（不带 --coverage）| ❌ threshold 不会被检查 |
| 没有抬高基线的流程 | ❌ 即使补了测试，基线永远是 0 |

### 12.2 三步把棘轮装上

#### Step 1：让 CI 跑覆盖率版本

修改 `.github/workflows/pr-gate.yml` 中 `frontend-unit-test` job 的最后一步：

```yaml
- name: Run unit tests with coverage
  working-directory: psa-front-desk-pc
  run: pnpm run test:unit:coverage

- name: Upload coverage report
  if: always()
  uses: actions/upload-artifact@v4
  with:
    name: pc-frontend-coverage
    path: psa-front-desk-pc/tests/unit/coverage
```

`test:unit:coverage` 已在 §4.2 的 scripts 里，跑时如果实际覆盖率 < `coverageThreshold`，Jest 会非零退出，job fail，PR 卡住。

#### Step 2：基线抬高节奏

每次有人补了一批新测试，覆盖率会自然涨。按以下节奏在 PR 内手动抬高 threshold：

```js
// jest.config.js
coverageThreshold: {
  global: { lines: 15, functions: 12, branches: 10, statements: 15 }
  // 改之前先本地跑 pnpm run test:unit:coverage 看真实值，
  // 抬到"略低于真实值 1-2 个百分点"，留点缓冲避免边缘 Flaky
}
```

**规则**：
- ✅ 只能调高，不能调低
- ✅ 调高必须与新增测试在同一个 PR
- ❌ 不允许"先调低再补测"
- ❌ 不允许"绕过 threshold"（不允许加 `--coverageThreshold='{}'` 之类的临时绕过）

#### Step 3：分模块阈值（可选，未来扩展）

随着覆盖率增长，全局阈值容易被"高覆盖大文件"拉高、掩盖"低覆盖关键文件"。届时改成分模块：

```js
coverageThreshold: {
  global: { lines: 30 },
  // 关键文件单独要求更高
  './src/utils/permission.js': { lines: 90, branches: 85 },
  './src/utils/ruoyi.js': { lines: 70 }
}
```

### 12.3 配套：基线抬高的 PR 模板

补测 PR 的 Test Plan 里加一条：

```markdown
## Test Plan

- [x] 本地 `pnpm run test:unit:coverage` 通过，新增覆盖率：lines 12% → 18%
- [x] `jest.config.js` 中 coverageThreshold 已从 0 调到 15（略低于真实值，留缓冲）
- [ ] CI `frontend-unit-test` job 通过（待 push 触发）
```

### 12.4 防止 Flaky 拖累棘轮

覆盖率本身偶发会因"同一行被多分支重复计算"或"动态加载文件被分片"波动 1-2%。处理：

- 阈值留 1-2% 缓冲（实际 16% 时阈值设 15，不要顶到 16）
- 如果某次 CI 因 0.5% 抖动失败，先调查是否真的回退；不是的话**不要**调低，而是补一条测试或修计算口径

## 13. 业务边界场景测试模板（PC 视角）

> **本章状态**：📋 模板章节。给 PC 端测试人员一份"补充视角"清单，每写新测试时对照检查。

### 13.1 时间类

| 边界 | 测试模板 |
|---|---|
| 跨月 / 跨年 | `parseTime` 处理 2025-12-31 23:59:59 → 2026-01-01 00:00:00 |
| 闰年 2/29 | 2024-02-29 不应被识别为 03-01；2025-02-29 应报错 |
| 夏令时切换日 | PSA prod 在中国（无夏令时），但**业务出差到欧美时**用户本地时间会有 1 小时漂移；前端展示应统一显示后端 UTC 转中国时间 |
| 时区漂移 | 用户跨时区出差时，工时申报 `worksDate` 应锁定**项目所在国家时区**，不是浏览器本地 |
| 毫秒 / 秒精度 | 后端 `approvedTime` 是 `DATETIME2(7)`，前端展示截到秒；批量比较时间时不能 `===` |
| 月末日 | 1 月 31 日加 1 个月应等于 2 月 28/29 日（业务定义），不是 3 月 3 日 |

### 13.2 数字精度类

| 边界 | 测试模板 |
|---|---|
| BigDecimal 序列化 | 后端返回 `0.30` 字符串，前端解析后展示必须是 `0.30` 不是 `0.3`（财务口径）|
| 浮点累加 | 0.1 + 0.2 在 JS 里是 0.30000000000000004，工时合计必须用整数（分钟）或字符串拼计算 |
| 整数溢出 | JS Number 安全整数上限 `2^53-1`，后端 Long 超过这个范围用字符串返回 |
| 除零 | 比率类计算（如绩效达成率 = 实际/目标），目标为 0 时显示 "-" 或 "N/A"，不是 "Infinity" / "NaN" |
| 负数 | 工时不应为负、金额可负（退款/冲抵），按字段语义区分 |

### 13.3 字符与编码类

| 边界 | 测试模板 |
|---|---|
| 中文全角空格 | 描述字段 trim 时全角空格 `　` 不被 `.trim()` 去掉 |
| Emoji（4 字节 UTF-8）| 描述字段长度限制按"字符"还是"字节"？数据库 `nvarchar(500)` 是字符 |
| HTML 转义 | 用户输入 `<script>` 直接渲染到 v-html 时会执行；只用 `{{}}` 插值默认安全 |
| URL 编码 | 查询参数含 `&`、`%`、`+` 时，前后端 url-encode 是否一致 |
| 大小写 | 用户名搜索是否大小写敏感？前端要不要 `.toLowerCase()` 比对 |
| 换行符 | `\n` vs `\r\n`（Windows 复制粘贴），描述字段保存后回显是否一致 |

### 13.4 集合与边界值

| 边界 | 测试模板 |
|---|---|
| 空集合 | 列表为空时 UI 显示"暂无数据"，不是无限 loading |
| 单元素 | 树形结构只有 1 个根节点时正常显示，不报"undefined.children" |
| 超大集合 | 列表 10000+ 行时是否分页、是否虚拟滚动、性能是否可接受 |
| 嵌套深度 | 组织树深度 5+ 层时菜单缩进不溢出 |
| 重复值 | 下拉选项有重名时按 id 区分，不按 name |

### 13.5 状态与并发

| 边界 | 测试模板 |
|---|---|
| 快速重复点击 | "提交"按钮在 ajax 期间应 disable，防止重复创建工时 |
| 网络丢包 / 超时 | axios 超时（默认 30s）应弹 toast 不应白屏 |
| token 过期 | API 返回 401 时统一拦截 → 清 token → 跳登录页 |
| 弹窗未关重复打开 | 同一弹窗多次触发应去重（Element UI 的 `$msgbox` 默认不去重，需要业务侧加 flag）|
| 浏览器后退/前进 | Vue Router 后退应正确恢复表单状态或明确提示"未保存内容" |
| 多标签页同步 | 标签页 A 退出登录后，标签页 B 下次请求应 401 跳登录 |
| 离线状态 | navigator.onLine = false 时 UI 应有降级展示 |

### 13.6 权限与角色

| 边界 | 测试模板 |
|---|---|
| 越权访问 | 员工账号直接访问 `/system/user` URL，应看到 403 或重定向到首页 |
| 越权数据 | 员工 A 在浏览器 URL 改 `?id=999` 查别人的工时，前端不显示 / 后端拒绝 |
| 角色切换 | 用户角色从员工升为 PM 后，刷新前后菜单是否一致 |
| 按钮可见性 vs 实际权限 | 不能只靠"按钮隐藏"防越权，按钮点击后端必须再次校验 |

### 13.7 接口数据形态（前端容错重灾区）

后端字段缺失、null、空集合是 PSA 历史 Bug 的高发源头。前端必须对所有"非自家计算出来的数据"做防御。

| 边界 | 测试模板 / 防御代码 |
|---|---|
| `response.data === null` | 后端"没数据"用 null 还是 `{}`？两边须约定。前端 `data?.field` 必须用可选链或默认值 |
| `response.data === undefined` | axios 拦截器返回 `undefined` 时（错误未抛干净）→ 前端 `data.list.map` 直接 TypeError |
| `response.data.list === []` | 空列表必须显示"暂无数据"提示，不是无限 loading 或空白 |
| `response.data.list === null` | 与 `[]` 行为应一致；调用方 `list?.map ?? []` 兜底 |
| 字段缺失 | 后端旧字段被废弃但前端还在用：`item.oldField` → undefined，渲染应不报错 |
| 字段类型漂移 | Long 字段在 Java 序列化时可能是 string（防 JS 精度丢失），前端比较时用 `String(a) === String(b)` |
| 数字 0 vs null | `if (count)` 把 0 也判为假；判存在用 `count != null` |
| 布尔字段反序 | 后端 0/1 vs 前端 true/false，axios 拦截层应统一映射 |
| 时间字段格式 | 后端可能返回 `'2026-05-08'` / `'2026-05-08 12:00:00'` / 时间戳数字，前端 `parseTime` 应都能解析 |
| 嵌套字段缺失 | `item.org?.name` 而非 `item.org.name`；常见于权限不足时后端剥离敏感字段 |
| 接口返回未定义字段 | 不能因新字段 `item.extraXxx` 出现就崩，TypeScript 严格模式下需放宽校验或加 .d.ts |
| 数组顺序不稳定 | 后端没显式 ORDER BY 时返回顺序可能变；前端必须自己排序，不依赖到达顺序 |
| 翻页边界 | `pageNo=999, total=5` 时后端可能返回空 list 也可能 400，前端都要兜底 |
| 401 / 403 / 500 | 拦截器应统一处理：401 → 清 token 跳登录；403 → toast "无权限"；500 → toast "服务异常"，不应让具体页面各自处理 |
| 超大响应体（10MB+）| dhtmlx-gantt / VXE 大列表分页拉取，避免一次性加载 |
| 重复请求 | 用户快速切换筛选时，先发的请求可能后返回，应用 race condition 模式（带 requestId 比对最新）|

**示例 spec：**

```js
// tests/unit/api/manhour-list.spec.js（示意）
describe('工时列表接口响应处理', () => {
  test('response.data.list = null → 视为空列表', () => {
    const normalized = normalizeManHourList({ data: { list: null, total: 0 } })
    expect(normalized.list).toEqual([])
    expect(normalized.total).toBe(0)
  })

  test('response.data.list 缺失 list 字段 → 视为空列表', () => {
    const normalized = normalizeManHourList({ data: {} })
    expect(normalized.list).toEqual([])
  })

  test('item.org 字段缺失（权限剥离）→ 显示"--"不崩溃', () => {
    const item = { id: 1, hours: 8 }   // 没有 org 字段
    expect(renderOrgName(item)).toBe('--')
  })

  test('Long ID 字段以字符串形式返回 → 比较前规范化', () => {
    const a = { id: '1234567890123456789' }   // 字符串
    const b = { id: 1234567890123456789 }     // 数字会精度丢失成 1234567890123456800
    expect(String(a.id) === String(b.id)).toBe(false)  // 警示：精度丢失不可比
  })
})
```

> 💡 **建议**：在 `src/utils/` 下补一个 `apiNormalize.js`，把"安全提取列表 / 安全提取分页 / 安全提取嵌套字段"封装成纯函数，然后给这些纯函数写单测。这样所有接口调用都过一层防御，单测覆盖率也高。

### 13.8 浏览器兼容性

PSA 客户端环境是 Brave / Chrome / Edge（基于 Chromium）。最低支持：

```markdown
P0 / P1 / P2 / P3
```

### 15.3 测试反馈环（防止同一 Bug 复发）

```
生产 Bug → 修复 PR → 新增防回归测试 → 进 CI 棘轮
    ↑                                              │
    └─── 如果再次发现同样 Bug，测试一定先红 ←────┘
```

### 16.3 后端 → 前端影响

### 16.5 怎么用这张表

**写 PR 描述时**：在"改动范围"小节，根据本表查到的影响范围，把"需要回归的测试"列入 Test Plan。

**Reviewer 看 PR 时**：对照本表检查"作者是否漏了关联回归"。

**测试人员（如本 Claude 角色）出测试方案时**：以本表为基础，主动追问"X 改动是否影响 Y"。

### 16.6 维护规则

- 出现一次"漏回归导致的生产 Bug" → 表里加一行
- 季度评审：对照实际 PR 改动看是否有新依赖路径
- 表项目过时（如某依赖已被解耦）→ 划掉但保留历史（用 `~~删除线~~`）
